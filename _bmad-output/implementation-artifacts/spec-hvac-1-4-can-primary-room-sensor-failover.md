## Intent

**Problem:** All 13 rooms' temperature/humidity control still resolves through `hvac/room_sensors.yaml`'s HA-primary/UDP-secondary failover, so even though HVAC-1.2/1.3 composed a working (publish-only) CAN sensor receiver onto `devices/climate-control.yaml`, no room's control-facing abstracted sensor ever prefers CAN, and the obsolete S1 UDP tier stays wired in.

**Approach:** Rewire `hvac/room_sensors.yaml`'s two `failover_sensor.yaml` inclusions to `primary_sensor: ${room_slug}_temp_can` / `${room_slug}_humidity_can` (the CAN source sensors HVAC-1.1-1.3 already publish) with `${room_slug}_temp_ha` / `${room_slug}_humidity_ha` as secondary, drop the now-unused `_temp_udp`/`_humidity_udp` sensor declarations, expose the abstracted sensors to Home Assistant, and extend `vesta/packages/components/failover_sensor.yaml` with optional, backward-compatible tier-label vars so the exposed tier text reads `CAN`/`HA`/`Emergency` instead of the generic `Primary`/`Secondary`/`Emergency`.

## Boundaries & Constraints

**Always:** Change `hvac/room_sensors.yaml` only for its `temperature_failover`/`humidity_failover` package vars (`primary_sensor`, `secondary_sensor`, `internal: "false"`, new tier-label vars) and its own sensor declarations — every one of the 13 room files under `hvac/rooms/**` keeps passing only `room_slug`, `room_name`, `ha_temperature_sensor_id`, `ha_humidity_sensor_id` unchanged, since `room_sensors.yaml` is their sole shared include. Preserve `${room_slug}_temp_abstracted`/`${room_slug}_humidity_abstracted` as the exact sensor IDs consumed by PID/dew-point/fancoil-boost/MEV-humidity logic. If extending `failover_sensor.yaml`, add only new optional `defaults:` vars whose default values reproduce today's literal `"Primary"`/`"Secondary"`/`"Emergency"` output exactly, so `vesta/tests/harness/failover_sensor.yaml` and `vesta/tests/e2e/test_failover_sensor.py` (which assert those literal strings and never set the new vars) keep passing unmodified. Update the doc comments in `hvac/room_sensors.yaml`, the Tier Priority Rules table there, `hvac/CLAUDE.md`'s entity-ID example block, and root `CLAUDE.md`'s failover-order bullets so none of them still describe HA-primary/UDP-secondary as current behavior.

**Block If:** Any room file under `hvac/rooms/**` is found passing additional vars to `room_sensors.yaml` beyond the four listed above, or any file outside `hvac/room_sensors.yaml`/`vesta/packages/components/failover_sensor.yaml` reads `${room_slug}_temp_udp` or `${room_slug}_humidity_udp` directly — re-verify before deleting those sensor declarations.

**Never:** Do not touch `devices/climate-control.yaml`'s `packet_transport: providers:` list or any `_co2`/`_iaq` `packet_transport` sensor in `hvac/rooms/**` (ground/first floor CO2 and IAQ still consume the same UDP providers — that is HVAC-1.5 scope). Do not regenerate `registry/nodes.csv` or `hvac/packages/generated/can_sensor_routes.yaml`, or otherwise resolve the pre-existing stale-registry gap already tracked in `deferred-work.md` — HVAC-1.4 wires the failover shape, it does not populate real CAN routes. Do not modify `hvac/mev_demand.yaml`, `hvac/mev_modbus.yaml`, `devices/gateway.yaml`, `hvac/packages/can_sensor_receiver.yaml`, `hvac/packages/generated/can_sensor_routes.yaml`, or any file under `canbus/protocol/`, `hvac/protocol/`, or `registry/`. Do not change the ESP_LOGx diagnostic message wording in `failover_sensor.yaml` beyond parameterizing the tier-name tokens already embedded via `%s`.

## Design Notes

`failover_sensor.yaml`'s lambda currently hardcodes tier-name string literals both for `current_tier`/`last_tier` bookkeeping and for the `%s`-templated diagnostic log lines (e.g. `ESP_LOGI("failover", "[%s] Failover: Primary → Secondary (primary sensor unavailable)", "${sensor_id}")`). The label vars must replace the literal in *every* place the same word appears (assignment, `last_tier == "..."` comparisons, and the two tier-name slots inside each log format string that currently read as English words rather than `%s` placeholders) — otherwise `last_tier` bookkeeping breaks when a caller overrides the labels. Callers that don't set the new vars (the harness, the vesta example, this repo's room files before this story) get byte-identical behavior since defaults reproduce the current literals.

The registry (`registry/nodes.csv`) and generated routes (`hvac/packages/generated/can_sensor_routes.yaml`) were out of sync with each other before this story (nodes.csv had a `sensors=1`/`soggiorno` row committed at HVAC-1.2 that was never regenerated into the routes file — tracked in `deferred-work.md`); resolving that sync is what made the HALT-and-fix in the Spec Change Log necessary. Post-fix: every room without a registered CAN node (12 of 13) has its `<room_slug>_temp_can`/`<room_slug>_humidity_can` declared but never published, so it reads NaN forever and falls through to HA — identical to the pre-story behavior. Soggiorno's node is registered, so its dispatch script is wired, but the physical CAN bus/hardware is still pre-live, so it also reads NaN today; the CAN-primary path becomes observable for any room only once real frames arrive on the bus.
