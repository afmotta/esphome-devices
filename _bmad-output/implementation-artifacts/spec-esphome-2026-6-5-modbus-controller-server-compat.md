## Intent

**Problem:** The repository currently documents and validates a minimum ESPHome version of 2026.5.0 for key controller paths, but the target support version is now 2026.6.5. This upgrade includes a Modbus architecture split where `modbus_controller` is client-only and slave-side behavior moves to `modbus_server`; in this repo we have many physical Modbus slave devices (relay/analog boards and MEV) but this does not automatically mean we are using ESPHome `modbus_server` YAML blocks.

**Approach:** Keep the change narrowly scoped to compatibility and support-version uplift: run an explicit audit that distinguishes physical slave devices from ESPHome server-mode declarations, migrate only if actual `modbus_server`/legacy server-mode YAML is found, and update version baselines plus docs to keep support claims consistent with validated behavior.

## Boundaries & Constraints

**Always:** Preserve current single-master Modbus topology and package contracts; make minimal, targeted edits; validate with ESPHome config/compile checks on relevant device entry points and test fixtures; keep secrets untouched; avoid behavioral changes unrelated to version compatibility.

**Ask First:** Any proposal that changes Modbus topology (for example introducing local slave exposure, changing controller/client ownership, or altering register contracts) or any broad refactor beyond version and compatibility adjustments.

**Never:** Introduce new control logic, rename established IDs/vars contracts, or perform speculative migration to `modbus_server` where no server-side component exists.

## I/O & Edge-Case Matrix

## Design Notes

This effort is a compatibility-hardening chore, not a feature delivery. The design intent is to prefer verification-first migration: prove whether the codebase actually uses server-side Modbus semantics before editing component declarations.

Golden rule for this migration:

```yaml
# Keep this pattern unchanged when acting as Modbus client/master
modbus_controller:
  - id: board_controller
    address: 0x02
    modbus_id: rs485_bus
```

Only apply `modbus_server` when a component is explicitly serving registers to external clients.
