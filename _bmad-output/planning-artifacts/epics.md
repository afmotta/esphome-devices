# canbus - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for the CAN Bus Smart Home PoC, decomposing the requirements from the PRD and Architecture into implementable stories.

## Requirements Inventory

### Functional Requirements

FR-1.1: Each node firmware SHALL detect 5 event types per button using ESPHome `on_multi_click`: single click, double click, triple click, long press (1–3 s), extra-long press (3+ s).
FR-1.2: Multi-click patterns SHALL be ordered longest-first (triple → double → single → long → extra-long) to prevent shadowing.
FR-1.3: Click timing thresholds are compile-time constants; runtime adjustment is out of scope.
FR-1.4: Each node SHALL support up to 6 buttons per board. PoC uses [ASSUMPTION: 2 buttons per node — confirm from bench setup].

FR-2.1: On each button event, the node SHALL transmit a CAN frame with CAN ID = `(CAT_INPUT << 9) | node_id` and the 8-byte button event payload.
FR-2.2: The node SHALL validate `x.size() >= CAN_FRAME_SIZE` in all CAN frame receive lambdas before indexing any byte.
FR-2.3: CAN bus speed SHALL be 125 kbps on all devices.
FR-2.4: The node SHALL use the MCP2515 CAN controller via SPI0 with a 16 MHz clock [ASSUMPTION: confirm oscillator from board docs — OQ-3].

FR-3.1: Each node SHALL transmit a heartbeat frame every 30 seconds with CAN ID = `(CAT_STATUS << 9) | node_id` and 8-byte heartbeat payload.
FR-3.2: Heartbeat error flags (payload byte 4) SHALL be set: bit 0 for TX error/timeout, bit 1 for bus-off; 0x00 when healthy.

FR-4.1: A shared C++ header `canbus_protocol.h` SHALL be authored from scratch as a PoC deliverable, defining all protocol constants and inline helper functions per the wire spec in this PRD. The existing exploration-generated file in `docs/esphome_canbus/` SHALL NOT be treated as correct or authoritative — it may be referenced but must be validated against the PRD spec and rewritten as needed.
FR-4.2: `canbus_protocol.h` SHALL define at minimum: `PROTO_V1`, `CAT_INPUT`, `CAT_STATUS`, `MSG_BUTTON_EVENT`, `MSG_HEARTBEAT`, all `EVT_*` event type constants, `CAN_FRAME_SIZE = 8`, `can_id()`, `button_payload()`, `event_type_str()`. All functions SHALL be `inline` to avoid ODR violations across multiple includes.
FR-4.3: All CAN frame construction and decoding in YAML lambdas SHALL use named constants and functions from `canbus_protocol.h`. Raw bit shifts, magic numbers, and magic hex values SHALL NOT appear in YAML lambdas.
FR-4.4: `canbus_protocol.h` SHALL be included by both node and gateway firmware and SHALL compile cleanly under both `rp2040` and `esp-idf` toolchains.

FR-5.1: The gateway SHALL receive all CAT_INPUT frames using mask-based filtering: `can_id: 0x200`, `can_id_mask: 0x600`.
FR-5.2: The gateway SHALL receive all CAT_STATUS frames using mask-based filtering: `can_id: 0x600`, `can_id_mask: 0x600`.
FR-5.3: The gateway CAN interface SHALL use the native TWAI controller on GPIO15 (TX) and GPIO16 (RX) at 125 kbps.

FR-6.1: On receiving a valid CAT_INPUT frame, the gateway SHALL fire an `esphome.canbus_button` event to HA with string fields: `room`, `board`, `button`, `event`.
FR-6.2: On receiving a valid CAT_STATUS frame, the gateway SHALL log the heartbeat frame via ESPHome logger (no HA event in PoC).
FR-6.3: All event data field values SHALL be strings. Integers converted via `to_string()` or `std::string()`.
FR-6.4: The gateway SHALL fire HA events via a single `lambda:` action using `api::global_api_server` directly — no ESPHome globals for event staging.

FR-7.1: The gateway SHALL communicate with Home Assistant using the ESPHome native API. The specific network layer (WiFi, Ethernet) is an implementation detail and may vary between PoC and production hardware.
FR-7.2: API credentials (key) SHALL be managed outside of committed source (e.g. `secrets.yaml`) and SHALL NOT be hardcoded in firmware configs.

FR-8.1: Node YAML files SHALL be generated by `generate_nodes.py` from `nodes.csv`. Hand-editing files in `firmware/nodes/` is prohibited.
FR-8.2: `nodes.csv` SHALL define at minimum: `node_id`, `floor`, `room`, `board`, `location`, `gpio_list` for each node.
FR-8.3: For the PoC, `nodes.csv` SHALL contain exactly 2 node entries (node IDs 100 and 101, ground floor range), with distinct `room` values and assigned button GPIO lists.
FR-8.4: `generate_nodes.py` SHALL print a CAN ID map on completion. The map SHALL be reviewed for duplicate IDs before flashing.

FR-9.1: For each of the 5 event types, pressing the button sequence on node 100 SHALL result in an `esphome.canbus_button` event in HA Developer Tools with correct `room`, `board`, `button`, and `event` values.
FR-9.2: The same validation SHALL pass for node 101 (different `room` value).
FR-9.3: Within 60 seconds of power-on, each node SHALL emit at least one heartbeat visible in ESPHome gateway logs.
FR-9.4: Events from node 100 and node 101 SHALL be distinguishable by `room` and `board` field values.
FR-9.5: All 10 event-type combinations (5 types × 2 nodes) SHALL produce correctly decoded HA events with no missing or blank fields.
FR-9.6: ESPHome logs on the gateway SHALL show zero CAN error frames during the validation run.

### NonFunctional Requirements

NFR-1 (Compile-first): Both node and gateway firmware SHALL compile successfully with `esphome compile` before any hardware is connected. A failed compile is a blocking defect.
NFR-2 (Lambda safety): Every CAN frame receive lambda SHALL guard with `if (x.size() < CAN_FRAME_SIZE) return;` before indexing. Absence of this guard is a blocking defect.
NFR-3 (Protocol SSoT): `canbus_protocol.h` is the sole location for protocol constants. Any constant duplicated in YAML is a defect.
NFR-4 (Framework): Gateway SHALL use `esp-idf` framework. Arduino framework is incompatible with `esp32_can` on ESP32-S3 and SHALL NOT be used on the gateway.
NFR-5 (Node isolation): Nodes SHALL have no WiFi, no ESPHome API, no OTA, no web server, no network logger.
NFR-6 (Bus speed uniformity): All devices on the bus SHALL use 125 kbps. Mismatched speeds are not recoverable without power cycling.
NFR-7 (Termination): The bus SHALL have exactly two 120 Ω termination resistors, one at each physical end.
NFR-8 (Observability): The gateway `on_frame` lambda SHALL produce ESPHome log output confirming each event decoded and fired. Silent failure in the chain is not acceptable.
NFR-9 (Repeatable validation): HA Developer Tools event log SHALL be captured as an artifact (screenshot or export) for each of the 10 acceptance combinations. Eyeball-only validation is not sufficient.

### Additional Requirements

- **Directory restructure (pre-compile gate):** All firmware must be moved from `docs/esphome_canbus/` to `firmware/` before any compile. All relative paths in YAML includes must be updated. This is Story 1.1.
- **Hardware open-question resolution (pre-compile gate):** OQ-1 (button GPIO numbers), OQ-2 (MCP2515 INT pin = GPIO20?), OQ-3 (16 MHz oscillator?) must be confirmed from board documentation before writing firmware. Any assumption that turns out to be wrong requires reflashing all nodes.
- **ESPHome version pinning:** Pin the ESPHome version immediately after first successful compile. Record the pinned version in `firmware/README.md`. All subsequent compiles use the pinned version.
- **Secrets management:** `firmware/secrets.yaml` (gitignored) holds `wifi_ssid`, `wifi_password`, `ha_api_key`. `firmware/secrets.yaml.example` is committed as a template.
- **Node ID allocation:** PoC nodes use IDs 100 and 101, from the ground floor range (100–199). `generate_nodes.py` SHALL reject IDs outside 0–399 or duplicates.
- **Single-lambda API pattern on gateway:** Home Assistant events are fired via `api::global_api_server` directly inside a single lambda. The exact C++ method signatures must be verified against the pinned ESPHome source before implementation.
- **Heartbeat PoC disposition:** Gateway logs heartbeat frames via `ESP_LOGI` only. No `esphome.canbus_heartbeat` HA event is fired in the PoC.
- **Commissioning procedure:** For each PoC node: edit `nodes.csv`, run generator, compile, flash, verify heartbeat in gateway log within 60 s before bench assembly.
- **PoC sign-off criteria:** (1) All 10 acceptance combinations produce correct HA events; (2) each event is captured as a logged artifact; (3) zero CAN error frames in gateway logs during the run.

### UX Design Requirements

N/A — This is an embedded IoT firmware project. There is no user interface.

**And** all functions are marked `inline` (no ODR violations across multiple includes)
**And** all constants use `UPPER_SNAKE_CASE`; all functions use `snake_case`
**And** the header includes a standard header guard (`#pragma once` or `#ifndef` guard)
**And** the header compiles without errors or warnings when included in a minimal ESPHome YAML compiled under the `rp2040` platform
**And** the header compiles without errors or warnings when included in a minimal ESPHome YAML compiled under the `esp32` / `esp-idf` platform
**And** no raw hex values, magic numbers, or bit shifts appear in any YAML lambda — all such logic lives exclusively in this header
