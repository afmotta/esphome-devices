- (decision) Framing adopted: one shared CAN infrastructure layer with Lighting and HVAC as application systems on top; Vesta (extractable library) and HA-side artifacts (cross-cutting generated deliverable) named as residents. Prevents: organizing by merge history (canbus/ = old-repo contents) which fuses transport + Lighting and leaves Lighting nameless.
- (decision) Scope: full physical restructure planned now (directory taxonomy decided and file moves mapped), landed as verifiable test-green slices. Prevents: conceptual-only drift where seams exist on paper but code keeps fusing.
- (decision) System boundaries live at the package/config layer; device entry-point configs compose packages across systems. Prevents: trying to map physical boards 1:1 to systems (gateway hosts infra+lighting; T-Connect swap may host lighting+HVAC).
- (constraint) HVAC-on-CAN is target-state, not present: devices/climate-control.yaml has zero canbus references today; map.json contract frozen (spec-map-json-contract) but consumer code not written. Restructure aligns to target without pretending the consumer exists.
- (constraint) canbus tooling is path-anchored (Path(__file__)) with push gate, generated nodes/, native+python tests; every move must keep generate_nodes.py byte-identical regeneration and tests green.
- (constraint) Pre-live on both subsystems: no migration shims, no compat layers; breaking moves made in place per standing project policy.
- (question) Registry ownership: nodes.csv (transport), bindings.yaml (lighting semantics), map.json+FLOOR_SLUGS (HVAC semantics) currently one system-of-record serving three layers - stays unified or splits per system?
- (decision) Registry: option C adopted - unified house registry elevated out of canbus/firmware/ to a TOP-LEVEL directory; one generator, one push gate, one manifest-hash mechanism (infra-owned); ownership is per-file with one file per system: nodes.csv+node_id_hwm (infra/transport), bindings.yaml (lighting), map.json (generated, HVAC-consumer contract frozen per spec-map-json-contract). Application-semantic validations are owned contributions (lighting owns bindings schema; HVAC owns map.json contract incl. room_slug/FLOOR_SLUGS). Prevents: split push gates / duplicated hash surfaces, and implicit nobody-owns-it files.
- (decision) Top-level taxonomy adopted: registry/ (house SoR), canbus/ (infra: protocol, node fw+packages, bridge, tools, tests, generated nodes stay inside), lighting/ (app: bindings semantics, fallback/relay packages, HA lighting automations), hvac/ (app: today's components/ - rooms, mev, sensors), vesta/ (library, unchanged), boards/ (shared hardware defs), devices/ (entry points composing packages across systems, incl locals/remotes variants), docs/, _bmad*. Systems own packages; devices/ owns composition. HVAC moves out of ambient root into hvac/. Prevents: new code defaulting into root; gateway being 'a canbus thing' when T-Connect will compose lighting+HVAC.
- (decision) HA-side artifacts split per system (Alberto's amendment): no monolithic home-assistant/ dir; each system exposes a home-assistant/ subfolder (lighting/home-assistant/ hold automations; canbus/home-assistant/ arbitration automations + generated manifest package; hvac/home-assistant/ dashboards). Discoverability preserved by the path convention '<system>/home-assistant/ is what HA imports'. generate_nodes.py output paths updated accordingly. Prevents: HA artifacts detached from the system whose behavior they encode.
- (constraint) remotes/*.yaml cite GitHub paths (devices/climate-control.yaml@main): entry-point moves must land atomically with the remotes that reference them (single commit, pre-live so acceptable).
- (decision) Arbitration is infra-owned and semantics-blind: canbus owns canonicalization, manifest hash, ha_ready and the fallback gate; lighting owns binding meaning (schema, ops, fan-out); contract surface between them is the compiled BindingEntry/bindings.h shape. Prevents: lighting schema edits silently breaking hash agreement; infra growing meaning-aware gate logic; extracting arbitration from lighting later if HVAC-on-CAN wants fallback.
- (decision) Cross-system contract rule: a boundary crossing systems is not a contract until it has (a) a frozen spec naming the fields and (b) a test that breaks when either side drifts (the spec-map-json-contract pattern generalized). Instances: map.json->HVAC (done), bindings->arbitration (spec+test due at carve-out), ADR-0006 sensor frames->HVAC (test due when consumer code is born).
- (decision) Conventions stay system-scoped: entity-ID conventions, per-system CLAUDE.md files (root CLAUDE.md becomes the map, not the rules), epic prefixes CAN- / LIGHT- / HVAC-. Historical canbus/_bmad-output stays frozen in place even though lighting history lives inside it.
- (assumption) devices/ = hand-maintained deployable entry points (climate-control, gateway+secrets, bridge, room sensors, locals/, remotes/); generated node firmware is exempt and stays in canbus/nodes/ next to its generator. Bridge qualifies as hand-maintained deployable even though it composes only canbus packages. canbus/firmware/ intermediate level flattened away once registry/gateway/bridge move out.
- (direction) Deferred: push-gate escalation (contract tests wired into check_registry_pushed); T-Connect PRO controller-swap decision (AD-4 composition shape is designed for it); HVAC-on-CAN consumer code placement; gateway package-extraction granularity; boards/ package unification gateway-vs-climate-master; Vesta extraction timing; scripts/ + secrets layout.
