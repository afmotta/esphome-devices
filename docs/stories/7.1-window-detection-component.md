# Story 7.1: Window Detection Component Foundation

**Epic:** Epic 7 - Window Detection & Climate Response  
**Story Type:** Component Foundation  
**Estimated Effort:** 1-2 days  
**Priority:** High (Foundation for Epic 7)

---

## User Story

**As a** climate control system developer,  
**I want** a reusable window detection component that integrates with Epic 5 emergency shutdown patterns,  
**So that** rooms can opt-in to equipment-aware climate shutdown when windows open.

---

## Story Context

### Existing System Integration

- **Integrates with:** Epic 5 emergency shutdown (`room_emergency_shutdown.yaml`), Epic 4 room components
- **Technology:** ESPHome YAML, state machine pattern, HA binary sensor integration
- **Follows pattern:** Epic 5 state machine (Normal → Emergency → Recovering), Epic 4 component composition
- **Touch points:** 
  - HA binary sensors for window state
  - PID climate controllers (force OFF when window open)
  - Diagnostic text sensors (add "Window Open" state)
  - Room component package inclusion pattern

### Key Design Decisions (from Brainstorming)

- **Equipment-aware:** Only affects equipment that wastes energy with open windows (fancoils)
- **Timeout-based:** 3-minute grace period before shutdown (allows brief ventilation)
- **Mode-aware:** Configurable `window_shutdown_modes` per room
- **Graceful degradation:** Window sensor unavailable = ignore, continue normal operation
- **State machine:** Normal → Window Open (180s) → Window Recovering (180s) → Normal

## Acceptance Criteria

### Functional Requirements

**FR1:** Component file `components/room_window_detection.yaml` created with complete vars interface:
- Required vars: `zone_slug`, `ha_window_sensor_id`, `window_shutdown_modes` (list)
- Optional vars: `window_timeout` (default 180s), `recovery_timeout` (default 180s)
- Component header documents vars contract with usage examples

**FR2:** State machine implemented with three states:
- **Normal:** Window closed OR window open < timeout, climate control operates normally
- **Window Open:** Window open > timeout, PID forced OFF mode
- **Window Recovering:** Window closed, stability period before resuming Normal

**FR3:** State transitions occur with correct timing:
- Normal → Window Open: Triggered after window open for 180 seconds (configurable)
- Window Open → Window Recovering: Triggered immediately when window closes
- Window Recovering → Normal: Triggered after 180 seconds of window remaining closed

**FR4:** PID integration forces climate control OFF in "Window Open" state:
- Component sets PID mode to OFF when entering "Window Open" state
- PID resumes previous mode when returning to "Normal" state
- Integration coordinates with existing room PID controllers

**FR5:** Mode-aware shutdown respects `window_shutdown_modes` configuration:
- If current climate mode not in `window_shutdown_modes`, ignore window state
- Example: Fancoil room with `shutdown_modes: [cooling]` only shuts down in cooling mode
- Example: Radiant room with `shutdown_modes: []` never shuts down

**FR6:** Diagnostic text sensor exposes current state to Home Assistant:
- Entity ID: `text_sensor.{zone_slug}_window_state`
- Possible values: "Normal", "Window Open", "Window Recovering"
- Updates in real-time as state transitions occur

**FR7:** Graceful degradation when window sensor unavailable:
- If `ha_window_sensor_id` state is "unavailable" or "unknown", treat as closed
- System continues normal operation relying on temperature sensors
- No errors logged, no disruption to climate control

### Integration Requirements

**IR1:** Component integrates with existing room component architecture (Epic 4):
- Can be included as package in room YAML files
- Uses vars substitution for zone-specific configuration
- Maintains separation from `room_sensors.yaml` (no bloat)

**IR2:** State machine coordinates with Epic 5 sensor emergency shutdown:
- Window state machine operates independently of sensor emergency state
- Both can be active simultaneously without conflicts
- Diagnostic sensor shows window state separately from sensor state

**IR3:** Existing room functionality continues unchanged when component not included:
- Rooms without window detection component operate exactly as before
- No breaking changes to existing room configurations
- Opt-in architecture prevents unintended side effects

### Quality Requirements

**QR1:** Component follows ESPHome YAML best practices:
- Uses native ESPHome components only (no custom C++)
- Proper indentation and structure
- Clear comments explaining state machine logic
- Vars defaults documented in component header

**QR2:** Component is self-documenting:
- Header includes purpose, required vars, optional vars, usage example
- State machine logic is clear and well-commented
- Integration requirements documented

**QR3:** Component file is complete and testable:
- Compiles without errors in ESPHome validation
- Can be tested standalone with mock room configuration
- No hardcoded values (all parameterized via vars)

## Technical Implementation Notes

### Component Structure

```yaml
# components/room_window_detection.yaml
# Purpose: Window-triggered climate shutdown for energy efficiency
# Epic: 7 - Window Detection & Climate Response
#
# Required vars:
#   - zone_slug: Entity ID prefix (e.g., "soggiorno")
#   - ha_window_sensor_id: HA binary sensor entity (e.g., "binary_sensor.soggiorno_window")
#   - window_shutdown_modes: List of modes that trigger shutdown (e.g., [cooling, heating] or [])
#
# Optional vars:
#   - window_timeout: Seconds before shutdown (default: 180)
#   - recovery_timeout: Seconds of stability before resuming (default: 180)
#
# Usage Example:
#   packages:
#     window_detection: !include
#       file: ../components/room_window_detection.yaml
#       vars:
#         zone_slug: soggiorno
#         ha_window_sensor_id: binary_sensor.soggiorno_window
#         window_shutdown_modes: [cooling, heating]

defaults:
  window_timeout: "180"
  recovery_timeout: "180"

# Binary sensor from HA (window state)
binary_sensor:
  - platform: homeassistant
    id: ${zone_slug}_window_sensor
    entity_id: ${ha_window_sensor_id}

# State machine implementation
# (Implementation details: delays, scripts, text sensor)
# See Epic 5 room_emergency_shutdown.yaml for state machine pattern reference
```

### State Machine Logic Reference

**Pattern to follow:** `components/room_emergency_shutdown.yaml`
- Uses delay actions for timeout handling
- Script-based state transitions
- Text sensor for state visibility
- Coordinates with PID controller via climate action calls

### Integration with PID

**Approach:** When entering "Window Open" state, call climate.turn_off action on PID
- PID entity ID: `climate.pid_${zone_slug}` (existing pattern from room components)
- Store previous mode to restore during recovery (or just resume with climate.turn_on)
- Coordinate with existing Epic 5 emergency shutdown (both can force OFF)

### Mode-Aware Logic

**Implementation:** Check current climate mode before triggering shutdown
- Use `id(climate.pid_${zone_slug}).mode` to get current mode
- Compare against `window_shutdown_modes` list
- Only transition to "Window Open" if mode matches

## Testing Scenarios

### Scenario 1: Normal Operation (Window Closed)
- **Given:** Window sensor reports "closed"
- **When:** Component is active
- **Then:** State is "Normal", PID operates normally

### Scenario 2: Window Open < Timeout
- **Given:** Window opens for 2 minutes then closes
- **When:** Timeout is 3 minutes
- **Then:** State remains "Normal", no shutdown occurs

### Scenario 3: Window Open > Timeout
- **Given:** Window opens and remains open for 4 minutes
- **When:** Timeout is 3 minutes
- **Then:** State transitions to "Window Open" at 3 minutes, PID forced OFF

### Scenario 4: Window Closes After Shutdown
- **Given:** System in "Window Open" state (PID OFF)
- **When:** Window closes
- **Then:** State transitions to "Window Recovering", remains OFF for 3 minutes, then returns to "Normal" and PID resumes

### Scenario 5: Mode-Aware (Not in Shutdown Modes)
- **Given:** Room configured with `shutdown_modes: [cooling]`
- **When:** Window opens while in heating mode
- **Then:** State remains "Normal", no shutdown (heating not in list)

### Scenario 6: Mode-Aware (In Shutdown Modes)
- **Given:** Room configured with `shutdown_modes: [cooling]`
- **When:** Window opens while in cooling mode for >3 minutes
- **Then:** State transitions to "Window Open", PID forced OFF

### Scenario 7: Window Sensor Unavailable
- **Given:** HA window sensor becomes "unavailable"
- **When:** Component is active
- **Then:** Treated as closed, state remains "Normal", climate control continues

### Scenario 8: Disabled Window Detection
- **Given:** Room configured with `shutdown_modes: []`
- **When:** Window opens
- **Then:** State remains "Normal", no shutdown occurs (disabled)

## Risk Assessment

### Primary Risks

**Risk 1: State machine conflicts with Epic 5 sensor emergency**
- **Impact:** Both state machines try to control PID simultaneously
- **Mitigation:** Design window state machine to coordinate (both can call climate.turn_off safely)
- **Validation:** Test with both window open AND sensor unavailable simultaneously

**Risk 2: Mode detection logic fails**
- **Impact:** Shutdown occurs in wrong modes or not at all
- **Mitigation:** Test extensively with mode changes before/during window open
- **Rollback:** Remove component from room package inclusion

**Risk 3: Timing issues with delays**
- **Impact:** State transitions occur too early or late
- **Mitigation:** Use ESPHome native delay actions (proven in Epic 5)
- **Validation:** Test with stopwatch to verify 180s timing

### Rollback Plan

**If component causes issues:**
1. Remove component inclusion from affected room YAML files
2. Redeploy room configuration without window detection
3. System returns to Epic 4 behavior (no window awareness)
4. No data loss, no configuration corruption
