# Story 2.2: Distribution Zone PID Simplification

## Story
**As a** distribution zone controller (distribuzione-piano-terra & distribuzione-primo-piano),  
**I want** to use a single PID climate entity per zone instead of dual heat/cool entities,  
**so that** the architecture is simplified, maintenance is easier, and the system follows the same pattern established in Story 2.1.

## Acceptance Criteria

1. **Pattern Consistency**: Apply the same single PID pattern successfully implemented in Story 2.1 to all distribution zones
2. **Piano Terra Zones Updated**: Replace `dual_pid.yaml` for all 4 ground floor zones:
   - Soggiorno (living room)
   - Cucina (kitchen)
   - Bagno (bathroom)
   - Anticamera (entryway)
3. **Primo Piano Zones Updated**: Replace `dual_pid.yaml` for both first floor zones:
   - Zona 1
   - Zona 2
4. **Entity ID Simplification**: Each zone has ONE climate entity instead of two:
   - Old: `climate.pid_{zone}_heat` + `climate.pid_{zone}_cool`
   - New: `climate.pid_{zone}` (supports both heat and cool modes)
5. **PID Tuning Preserved**: Maintain all existing PID parameters (Kp, Ki, Kd) for all zones in both modes
6. **Sensor Integration Maintained**: Keep existing sensor integrations:
   - Piano Terra: Room temperature sensors via `{zone}_room_temp_abstracted`
   - Primo Piano: Failover sensor via `termometro_primo_piano_temperature_abstracted`
7. **Output Control Unchanged**: Maintain slow_pwm output control for all zone relays
8. **Mode Coordination**: Implement global `climate_mode` coordination (similar to Story 2.1 approach)
9. **Validation Success**: Configuration validates and firmware compiles for both devices without errors
10. **HA Migration Documentation**: Document entity ID mapping for all 6 zones for Home Assistant migration

## Dev Notes

### Context from Story 2.1

Story 2.1 successfully simplified the mixing valve PIDs in `gruppo-miscelazione.yaml` by:
- Replacing `mixing_valve.yaml` + `dual_pid.yaml` pattern with direct single PID climate configuration
- Using `heat_output` and `cool_output` parameters (both pointing to same DAC output)
- Implementing mode coordination via `climate_mode` text sensor from Home Assistant
- Adding relay control via `on_state` automation (turn on when active, off when idle)
- Exposing PID output sensors for Home Assistant monitoring

**Key Pattern from Story 2.1:**
```yaml
climate:
  - platform: pid
    id: pid_${circuit}
    name: "PID ${circuit_name}"
    sensor: ${sensor_id}
    default_target_temperature: ${temp}
    heat_output: ${output}
    cool_output: ${output}
    control_parameters:
      kp: ${kp}
      ki: ${ki}
      kd: ${kd}
    on_state:
      - lambda: |-
          if (x.mode == CLIMATE_MODE_OFF) {
            id(${relay}).turn_off();
          } else {
            id(${relay}).turn_on();
          }
```

### Distribution Zone Specifics

**Piano Terra (distribuzione-piano-terra.yaml):**
- 4 zones with individual room temperature sensors via Modbus
- Sensors: `{zone}_room_temp_abstracted` (e.g., `soggiorno_room_temp_abstracted`)
- Outputs: `slow_pwm_{zone}` (e.g., `slow_pwm_soggiorno`)
- PID Tuning: kp=0.8, ki=0.005, kd=0.05 (room temp control - slower response)
- Default targets: 22°C (except Bagno=23°C, Anticamera=20°C)
- Relay control: via `remote_switch` template + `switch_management` script

**Primo Piano (distribuzione-primo-piano.yaml):**
- 2 zones sharing failover temperature sensor
- Sensor: `termometro_primo_piano_temperature_abstracted`
- Outputs: `slow_pwm_zona_1`, `slow_pwm_zona_2`
- PID Tuning: kp=0.2, ki=0.01, kd=0.0 (supply temp control - faster response)
- Default targets: 28°C (supply water temp, not room temp)
- Relay control: via `remote_switch` template + `switch_management` script

**Key Differences from Mixing Valves (Story 2.1):**
1. **Outputs**: slow_pwm instead of DAC outputs
2. **Relay Control**: Uses template switch + script instead of direct relay control
3. **Sensor Types**: Room temp sensors (Piano Terra) vs supply temp sensor (Primo Piano)
4. **PID Tuning**: Different parameters based on control strategy (room vs supply temp)

## Risk Assessment

### Risks
1. **Breaking HA automations**: Entity ID changes affect 6 zones across 2 boards
2. **Different control strategies**: Piano Terra (room temp) vs Primo Piano (supply temp) may need different relay coordination
3. **Relay/PWM coordination**: slow_pwm outputs with template switch may behave differently than DAC outputs
4. **Multi-zone interference**: Multiple zones on same board need independent control validation
5. **Sensor failover complexity**: Piano Terra has per-zone sensors; Primo Piano shares one sensor between zones

### Mitigation
1. **Comprehensive entity ID mapping** documented before deployment
2. **Preserve exact PID tuning parameters** for each zone's control strategy
3. **Test relay control** thoroughly - verify slow_pwm + remote_switch + switch_management integration
4. **Zone independence testing** - ensure one zone doesn't affect others
5. **Failover testing** - verify sensor_failover logic still works correctly with single PID

### PID Parameter Preservation - Critical
**Piano Terra zones** use room temperature control:
- Soggiorno, Cucina: target=22°C, kp=0.8, ki=0.005, kd=0.05
- Bagno: target=23°C, kp=0.8, ki=0.005, kd=0.05
- Anticamera: target=20°C, kp=0.8, ki=0.005, kd=0.05

**Primo Piano zones** use supply temperature control:
- Zona 1, Zona 2: target=28°C, kp=0.2, ki=0.01, kd=0.0

These parameters MUST be preserved exactly - they are tuned for different control strategies.

### Mode Coordination Strategy
Similar to Story 2.1, use `text_sensor.climate_mode` from Home Assistant to coordinate all zone PIDs:
- When climate_mode = "HEAT" → all PIDs switch to CLIMATE_MODE_HEAT
- When climate_mode = "COOL" → all PIDs switch to CLIMATE_MODE_COOL
- When climate_mode = "OFF" → all PIDs switch to CLIMATE_MODE_OFF

Implement as single automation that updates all 6 zone PIDs simultaneously.

### Relay Control Complexity Note
Unlike Story 2.1 (direct relay control), distribution zones use:
1. `slow_pwm` output platform
2. `remote_switch` template switch
3. `switch_management` script

Ensure the PID `on_state` automation correctly interfaces with this relay control chain.
