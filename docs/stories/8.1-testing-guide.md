## Test Setup

### Option 1: Standalone Test Device

Create a minimal ESP32 configuration that includes only the emergency condition component:

```yaml
# File: test-emergency-condition.yaml
esphome:
  name: test-emergency-condition

esp32:
  board: esp32dev
  framework:
    type: arduino

wifi:
  ssid: !secret wifi_ssid
  password: !secret wifi_password

api:

logger:
  level: DEBUG

# Include emergency condition component
packages:
  emergency_condition:
    url: !secret packages_config
    file: components/room_emergency_condition.yaml
    vars:
      zone_slug: "test_room"
      zone_name: "Test Room"
      ha_temperature_sensor_id: "sensor.test_room_temperature"
      emergency_timeout: 60  # Shortened for testing (1 min instead of 3 min)
      recovery_timeout: 30   # Shortened for testing (30s instead of 60s)
```

## Test Scenarios

### Test 1: Normal State (Default)

**Objective:** Verify component initializes in Normal state with priority=1

**Procedure:**
1. Deploy test configuration to ESP32
2. Monitor logs: `esphome logs test-emergency-condition.yaml`
3. Wait 30 seconds for initialization

**Expected Results:**
- Global `test_room_emergency_state` = 0 (Normal)
- Global `test_room_emergency_priority` = 1
- Text sensor `test_room_emergency_state_sensor` = "Normal"
- Logs show: `"Test Room: EMERGENCY NORMAL - Sensor healthy"`

**Pass Criteria:**
- ✅ State global is 0
- ✅ Priority global is 1
- ✅ Text sensor shows "Normal"

---

### Test 2: Normal → Active Transition

**Objective:** Verify component transitions to Active after sensor unavailable for emergency_timeout

**Procedure:**
1. Start in Normal state (Test 1 passed)
2. In Home Assistant: Disconnect the temperature sensor entity (e.g., disable integration)
3. Monitor logs and wait for emergency_timeout (60s in test config)
4. Check global state and text sensor

**Expected Results:**
- After 60 seconds:
  - Global `test_room_emergency_state` = 1 (Active)
  - Text sensor `test_room_emergency_state_sensor` = "Active"
  - Logs show: `"Test Room: EMERGENCY ACTIVE - Sensor unavailable for 60 seconds"`
  - Global `test_room_emergency_last_trigger_time` is non-zero

**Pass Criteria:**
- ✅ State transitions from 0 → 1 after timeout
- ✅ Text sensor shows "Active"
- ✅ Log message confirms transition
- ✅ Last trigger timestamp recorded

---

### Test 3: Active → Recovering Transition

**Objective:** Verify component transitions to Recovering when sensor restored

**Procedure:**
1. Start in Active state (Test 2 passed)
2. In Home Assistant: Reconnect the temperature sensor entity (re-enable integration)
3. Wait 10-15 seconds for next state machine check interval
4. Check global state and text sensor

**Expected Results:**
- Within 10-15 seconds:
  - Global `test_room_emergency_state` = 2 (Recovering)
  - Text sensor `test_room_emergency_state_sensor` = "Recovering"
  - Logs show: `"Test Room: EMERGENCY RECOVERING - Sensor restored"`

**Pass Criteria:**
- ✅ State transitions from 1 → 2 immediately when sensor valid
- ✅ Text sensor shows "Recovering"
- ✅ Log message confirms transition

---

### Test 4: Recovering → Normal Transition

**Objective:** Verify component transitions to Normal after recovery_timeout stability period

**Procedure:**
1. Start in Recovering state (Test 3 passed)
2. Keep sensor connected and stable
3. Wait for recovery_timeout (30s in test config)
4. Check global state and text sensor

**Expected Results:**
- After 30 seconds:
  - Global `test_room_emergency_state` = 0 (Normal)
  - Text sensor `test_room_emergency_state_sensor` = "Normal"
  - Logs show: `"Test Room: EMERGENCY NORMAL - Recovery complete after 30 seconds"`

**Pass Criteria:**
- ✅ State transitions from 2 → 0 after recovery timeout
- ✅ Text sensor shows "Normal"
- ✅ Log message confirms transition with duration

---

### Test 5: Recovering → Active Re-trigger

**Objective:** Verify component re-triggers to Active if sensor fails during recovery

**Procedure:**
1. Start in Active state (Test 2 passed)
2. Reconnect sensor to enter Recovering state (Test 3)
3. Wait 10 seconds (less than recovery_timeout of 30s)
4. Disconnect sensor again to simulate sensor flapping
5. Wait 10-15 seconds for state machine check
6. Check global state and text sensor

**Expected Results:**
- Within 10-15 seconds after re-disconnect:
  - Global `test_room_emergency_state` = 1 (Active)
  - Text sensor `test_room_emergency_state_sensor` = "Active"
  - Logs show: `"Test Room: EMERGENCY RE-TRIGGERED - Sensor lost again during recovery"`

**Pass Criteria:**
- ✅ State transitions from 2 → 1 when sensor fails during recovery
- ✅ Text sensor shows "Active"
- ✅ Log message indicates re-trigger
- ✅ Recovery timer reset

---

### Test 6: Pre-Emergency Warning

**Objective:** Verify component logs warnings before emergency triggers

**Procedure:**
1. Start in Normal state
2. Disconnect sensor
3. Monitor logs during first 60 seconds (before emergency_timeout)

**Expected Results:**
- Every 10 seconds (state machine interval):
  - Logs show: `"Test Room: WARNING - No sensor data for X / 60 seconds"`
  - State remains 0 (Normal)
  - Text sensor remains "Normal"

**Pass Criteria:**
- ✅ Warning logs appear every 10s
- ✅ State remains Normal until timeout exceeded
- ✅ Countdown shows progress toward timeout

## Interface Contract Validation

Use this checklist to verify the component follows the Epic 8 interface contract v1.0:

### Required Elements

- [x] **State global** defined: `${zone_slug}_emergency_state`
- [x] **State global** type is `int`
- [x] **State global** initial value is `"0"`
- [x] **Priority global** defined: `${zone_slug}_emergency_priority`
- [x] **Priority global** type is `int`
- [x] **Priority global** value is `1` (highest priority)
- [x] **Text sensor** defined showing state as "Normal"/"Active"/"Recovering"
- [x] **Text sensor** update interval is 5s (≤ 10s)

### State Machine Behavior

- [ ] Condition transitions to **Active (1)** when trigger persists beyond timeout (Test 2)
- [ ] Condition transitions to **Recovering (2)** when trigger clears from Active (Test 3)
- [ ] Condition transitions to **Normal (0)** after recovery stability period (Test 4)
- [ ] State updates within 10 seconds of transition trigger (all tests)
- [ ] No direct Active → Normal transition (must pass through Recovering) (verified in tests)

### Logging & Observability

- [ ] Log messages at state transitions (INFO/WARN level) (all tests)
- [ ] Log messages include zone name and new state (all tests)
- [x] Optional: Last trigger timestamp tracked
- [ ] Global values observable in Home Assistant dashboard

### Documentation

- [x] Component header documents required vars
- [x] Component header references interface spec v1.0
- [ ] Example usage provided (this testing guide)

## Debugging Tips

**Component doesn't load:**
- Check ESPHome compilation errors: `esphome config test-emergency-condition.yaml`
- Verify all required vars are provided in packages section
- Check YAML indentation (use 2 spaces, not tabs)

**State not transitioning:**
- Check logs for state machine interval running (every 10s)
- Verify HA sensor entity ID is correct
- Check if sensor entity exists in Home Assistant
- Verify ESPHome API connection to Home Assistant

**Text sensor shows "Unknown":**
- Check if state global has invalid value (not 0/1/2)
- Review state machine logic for edge cases
- Check logs for state transition messages

**Timestamp not updating:**
- Verify `millis()` function available in ESPHome
- Check if last_trigger_time global exists
- Confirm trigger event actually occurred (state = 1)
