## Status

OBSOLETE - Native ESPHome packet_transport diagnostics and Story 9.3 tier sensor provide sufficient monitoring

---

## Obsolescence Rationale

This story proposed custom UDP diagnostic sensors for sender and receiver components. However, multiple factors make this implementation redundant:

**1. Story 9.2 Obsolete (No Custom Components)**
- Story 9.2 (custom UDP receiver) was marked OBSOLETE - ESPHome's native `packet_transport` platform provides receiver functionality
- Story 9.1 used inline UDP configuration, not a separate `udp_sender.yaml` component
- Tasks 1-2 reference non-existent component files (`udp_sender.yaml`, `udp_receiver.yaml`)

**2. Native ESPHome Diagnostics Already Available**
ESPHome's `packet_transport` platform provides built-in connection status monitoring:
```yaml
binary_sensor:
  - platform: packet_transport
    provider: device-name
    type: status
    name: "Device Connection Status"
```

This automatically tracks:
- Connection state (connected/disconnected)
- Last packet received timestamp (via ping-pong feature)
- Provider availability

**3. Story 9.3 Already Provides Key Diagnostics**
The `sensor_failover_3tier.yaml` component created in Story 9.3 includes:
- `text_sensor.{zone}_sensor_tier` showing "UDP", "HA", or "Emergency"
- Comprehensive logging on tier transitions with reasons
- Real-time failover state visibility

**4. ESPHome Built-in Logging**
ESPHome provides extensive logging for packet_transport:
- Packet send/receive events logged automatically
- Parse errors logged by the platform
- No custom counter sensors needed for debugging

**What Might Still Be Valuable (Future Work)**
- HA dashboard template (Task 5) - Could be useful for visualization, but doesn't require custom sensors
- Troubleshooting guide (Task 6) - Documentation is always helpful, could be standalone doc

**Recommendation:** Use native ESPHome diagnostics + Story 9.3 tier sensor. If additional monitoring is needed, create a lightweight documentation story for HA dashboard templates and troubleshooting guides.

## Story

**As a** system operator,  
**I want** comprehensive UDP diagnostic sensors on all boards,  
**so that** I can monitor UDP health, troubleshoot network issues, and understand failover behavior through the Home Assistant dashboard.

---

## Acceptance Criteria

1. Master board (gruppo-miscelazione) exposes UDP sender diagnostic sensors
2. Slave boards (distribuzione boards) expose UDP receiver diagnostic sensors
3. All diagnostic sensors are visible in Home Assistant dashboard
4. Master diagnostics include: `udp_packets_sent` (counter), `udp_sender_status` (text), `udp_last_broadcast` (timestamp)
5. Slave diagnostics include: `udp_packets_received` (counter), `udp_parse_errors` (counter), `udp_last_received` (timestamp), `udp_receiver_status` (text)
6. Diagnostic sensors update in real-time (within 10-second update interval)
7. Text status sensors show human-readable states: "Enabled", "Disabled", "Error", "No Data"
8. Diagnostic sensors persist through board reboots (use `restore_value: true` where appropriate)
9. HA dashboard template created showing UDP health for all 3 boards
10. Troubleshooting guide document created mapping diagnostic values to resolution steps

### Diagnostic Sensor Implementation

**Master Board (UDP Sender):**
```yaml
sensor:
  - platform: template
    id: udp_packets_sent
    name: "UDP Packets Sent"
    accuracy_decimals: 0
    state_class: total_increasing
    restore_value: true
    
  - platform: template
    id: udp_last_broadcast
    name: "UDP Last Broadcast"
    device_class: timestamp
    
text_sensor:
  - platform: template
    id: udp_sender_status
    name: "UDP Sender Status"
    lambda: |-
      if (${use_udp}) {
        return {"Enabled"};
      } else {
        return {"Disabled"};
      }
```

**Slave Board (UDP Receiver):**
```yaml
sensor:
  - platform: template
    id: udp_packets_received
    name: "UDP Packets Received"
    accuracy_decimals: 0
    state_class: total_increasing
    restore_value: true
    
  - platform: template
    id: udp_parse_errors
    name: "UDP Parse Errors"
    accuracy_decimals: 0
    state_class: total_increasing
    restore_value: true
    
  - platform: template
    id: udp_last_received
    name: "UDP Last Received"
    device_class: timestamp

text_sensor:
  - platform: template
    id: udp_receiver_status
    name: "UDP Receiver Status"
    lambda: |-
      if (-5{use_udp}) return {"Disabled"};
      
      // Check if we've received packets recently (within 60s)
      auto last_rcv = id(udp_last_received).state;
      if (isnan(last_rcv) || (millis() - last_rcv > 60000)) {
        return {"No Data"};
      }
      
      // Check for excessive parse errors
      if (id(udp_parse_errors).state > 10) {
        return {"Error"};
      }
      
      return {"Enabled"};
```

### Troubleshooting Guide Structure

**Document Outline (`docs/epic-9-udp-troubleshooting.md`):**

1. **Diagnostic Sensor Reference**
   - Table mapping each sensor to its meaning
   - Normal vs. abnormal values

2. **Common Issues and Resolution**
   - "No Data" on slave board → Check master board is broadcasting, check network connectivity
   - High parse errors → Check JSON format, check network packet corruption
   - Sender/receiver "Disabled" → Check `use_udp` feature flag

3. **Network Debugging Tools**
   - Wireshark capture commands
   - tcpdump packet inspection
   - Ping and network connectivity tests

4. **Failover Tier Interpretation**
   - "UDP" tier → Normal, using UDP
   - "HA" tier → Degraded, UDP unavailable but HA working
   - "Emergency" tier → Critical, both UDP and HA unavailable

5. **Recovery Procedures**
   - How to force failover to HA tier (disable UDP)
   - How to restart UDP communication
   - When to rollback to Epic 8 architecture

### Testing Standards

**Diagnostic Sensor Validation:**
1. **Counter Test:** Verify counters increment correctly, persist across reboots
2. **Timestamp Test:** Verify timestamps update on events, show correct time
3. **Status Test:** Verify status transitions: Enabled → No Data → Enabled
4. **Dashboard Test:** Verify all entities appear in HA, update in real-time

**Integration Test Scenarios:**
1. Normal operation → All sensors show healthy values
2. Master board offline → Slave status shows "No Data", tier switches to "HA"
3. Network issue → Slave parse errors increase, status shows "Error"
4. Master board returns → Slave receives packets, tier switches back to "UDP"
