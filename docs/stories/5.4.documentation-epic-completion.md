## Troubleshooting Guide

**Common Issues and Solutions:**

### Issue 1: Room Stuck in Emergency State
**Symptoms:** `text_sensor.{zone}_sensor_state` shows "Emergency", won't clear  
**Causes:**
- HA sensor entity doesn't exist or is unavailable
- Entity ID mismatch in room component vars
- HA not providing valid sensor data

**Solutions:**
1. Check HA sensor entity exists: `sensor.room_{zone}_temperature`
2. Verify entity ID matches room component `ha_temperature_sensor_id` var
3. Check HA sensor has valid state (not `unavailable` or `unknown`)
4. Review ESPHome logs for error messages

### Issue 2: Emergency Not Triggering When Expected
**Symptoms:** HA disconnected >3 min, but system still operating  
**Causes:**
- `emergency_timeout` set too high
- HA sensor caching old value (appears valid but stale)
- Lambda logic error

**Solutions:**
1. Verify `emergency_timeout` var is 180s (3 min)
2. Check `sensor.{zone}_room_sensor_last_update` timestamp
3. Review lambda code for timeout calculation errors
4. Test with forced HA disconnection (not just sensor unavailability)

### Issue 3: Recovery Takes Too Long
**Symptoms:** HA reconnected, but system stays in "Recovering" state indefinitely  
**Causes:**
- `recovery_stability_timeout` set too high
- HA sensor flapping (connects, disconnects, repeats)
- Recovery logic never reaching stability threshold

**Solutions:**
1. Check `recovery_stability_timeout` var (default 60s)
2. Monitor HA sensor for stability (must be valid continuously for 60s)
3. Check network stability (WiFi, Ethernet connection quality)
4. Review logs for repeated "recovery started" messages

### Issue 4: Relay Not Turning Off During Emergency
**Symptoms:** Emergency triggered, but physical relay still energized  
**Causes:**
- Emergency action lambda not calling relay switch
- Relay switch ID incorrect
- Hardware issue (relay stuck)

**Solutions:**
1. Verify emergency lambda includes `switch.turn_off: relay_X`
2. Check relay switch ID matches physical wiring
3. Test relay manually via HA (can you turn it off?)
4. Check physical relay wiring and operation

### Issue 5: PID Not Resuming After Recovery
**Symptoms:** System exits emergency, but PID stays OFF  
**Causes:**
- Recovery logic not setting PID mode
- PID setpoint lost during emergency
- HA not sending mode/setpoint commands

**Solutions:**
1. Verify recovery lambda sets PID mode (not just allows it)
2. Check if PID setpoint was saved before emergency
3. Review HA automations that control PID mode
4. Manually set PID mode via HA to verify it works

## Performance Metrics

**Before Epic 5 (Modbus Sensors):**
- Modbus sensor devices: 16+ (across 3 ESPHome nodes)
- RS485 wiring: Complex, prone to noise issues
- Sensor polling: 30s interval via Modbus master
- Cost per sensor: ~$30-50 USD
- `room_sensors.yaml`: 221 lines of code
- Entity count per room: 7 entities (temp_modbus, humidity_modbus, temp, humidity, abstracted, online, last_update)

**After Epic 5 (HA-Only Sensors):**
- Modbus sensor devices: 0 (eliminated)
- RS485 wiring: None for sensors (only for outputs)
- Sensor polling: Direct from HA (Zigbee/WiFi sensors)
- Cost per sensor: Leverage existing HA sensors ($0 incremental)
- `room_sensors.yaml`: ~100 lines of code (60% reduction)
- Entity count per room: 3 entities (abstracted, state, last_update)

**Cost Savings:**
- Hardware: 16 sensors × $40 avg = $640 saved
- Wiring labor: Estimated 8-12 hours saved
- Maintenance: Simpler architecture, fewer failure points

**Code Complexity:**
- Component lines: 221 → 100 (54% reduction)
- Failover tiers: 3 → 2 (simpler logic)
- Required vars: 5 → 3 (easier configuration)
