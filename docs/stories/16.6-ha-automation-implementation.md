## User Story

As a **climate control system**,  
I want **a main automation that controls MEV fan speed based on the final fan demand**,  
So that **fan speed modulates continuously and responds to both air quality and humidity**.

---

## Story Context

This is the central fan speed control automation that ties all the pieces together:

- **Story 16.1:** Provides `sensor.first_floor_max_humidity`
- **Story 16.2:** Provides `sensor.mev_air_quality_demand` (CO₂/IAQ → fan demand)
- **Story 16.3:** Provides `sensor.first_floor_humidity_rate` (rate of change)
- **Story 16.4:** Provides state machine automations (escalate/de-escalate equipment)
- **Story 16.5:** Provides `sensor.mev_final_fan_speed` (max of AQ and humidity demands)

This automation reads `sensor.mev_final_fan_speed` and commands `number.first_floor_mev_fan_speed`.

**State Machine Automations (Story 16.4) handle equipment:**
- Dehumidifier → `switch.first_floor_mev_dehumidifier`
- Cooling pump → `switch.first_floor_mev_cooling`

## Acceptance Criteria

### Functional Requirements

1. **Fan Speed Control Automation:**
   - [ ] Trigger on `sensor.mev_final_fan_speed` state change
   - [ ] Trigger on `input_select.mev_humidity_state` state change (for immediate response)
   - [ ] Trigger periodically (every 30 seconds) as fallback
   - [ ] Command `number.first_floor_mev_fan_speed` with the final fan speed value

2. **Restart Mode:**
   - [ ] Use `mode: restart` to allow rapid updates without queuing

3. **Error Handling:**
   - [ ] Default to minimum fan speed (20%) if sensor unavailable
   - [ ] Fan speed = max(90%, AQ demand)
   - [ ] Dehumidifier ON
   - [ ] Cooling pump ON (if Summer Mode AND min_off idle)

6. **Pump Protection:**
   - [ ] Start only if min_off timer is idle
   - [ ] Stop only if min_on timer is idle
### Technical Requirements

4. **Equipment Control:**
   - [ ] Equipment (dehumidifier, cooling) is controlled by Story 16.4 state machine automations
   - [ ] This automation ONLY controls fan speed

5. **Automation mode:** `restart` (new triggers restart the automation)
6. **Error handling:** Graceful if any entity unavailable

## Technical Implementation

### Main Fan Speed Control Automation

```yaml
automation:
  - alias: "MEV Fan Speed Control"
    id: mev_fan_speed_control
    description: "Continuously modulates MEV fan speed based on air quality and humidity"
    mode: restart
    
    trigger:
      # Final fan speed changes
      - platform: state
        entity_id:
          - sensor.mev_final_fan_speed
          - input_select.mev_humidity_state
      # Periodic fallback (ensures state is correct even if triggers missed)
      - platform: time_pattern
        seconds: "/30"
    
    action:
      - service: number.set_value
        target:
          entity_id: number.first_floor_mev_fan_speed
        data:
          value: "{{ states('sensor.mev_final_fan_speed') | int(20) }}"
```

**That's it!** Equipment control (dehumidifier, cooling) is handled by the state machine automations in Story 16.4.

## Notes

- **Simplicity:** This automation is intentionally minimal—just reads a sensor and sets a number. All the complexity is in the template sensors and state machine logic.
- **Separation of Concerns:** Fan speed is continuous modulation; equipment (dehumidifier, cooling) is discrete state transitions. They are handled by separate automations.
- **Restart Mode:** Allows rapid updates. If a new fan speed is calculated before the previous action completes, it restarts with the new value.
- **30-second fallback:** Ensures the system recovers if triggers are missed or sensors briefly unavailable.
