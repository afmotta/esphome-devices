# Story 3.2: Remove Modbus Slave from distribuzione-piano-terra

## Status
Ready for Review

## Story
**As a** system architect,  
**I want** to remove Modbus slave functionality from distribuzione-piano-terra and replace remote sensor reading with direct Home Assistant sensor integration,  
**so that** the distribution board operates independently without requiring board-to-board Modbus communication.

## Acceptance Criteria

1. Remove `modbus_slave` package from `devices/distribuzione-piano-terra.yaml`
2. Remove `modbus_slave_registers` package from `devices/distribuzione-piano-terra.yaml`
3. Replace `sensor_failover` logic with direct Home Assistant sensor subscriptions
4. Verify all fancoils continue to operate correctly with HA-provided temperatures
5. Configuration validates and compiles without errors
6. No regression in fancoil temperature control or climate mode handling

### Architecture Context
**Current Role**: distribuzione-piano-terra is a Modbus slave board (A16 Ethernet) that:
- Reads Dallas temperature sensors from gruppo-miscelazione via Modbus registers
- Uses sensor_failover to handle Modbus disconnections with local fallback sensors
- Controls 8 heating zones on the ground floor

**After Epic 3.2**: distribuzione-piano-terra becomes independent board that:
- Subscribes to Dallas sensors from gruppo-miscelazione via Home Assistant
- No Modbus slave functionality
- Home Assistant coordinates climate modes and temperatures

### Key Preservation Points
1. **Fancoil PIDs**: Keep PID configurations, only change sensor source
2. **Relay/Output Control**: No changes to fancoil relay/valve outputs
3. **ESPHome API**: Ensure enabled for bidirectional HA integration

### Home Assistant Sensor Integration Pattern
```yaml
sensor:
  - platform: homeassistant
    id: piano_terra_temperature
    entity_id: sensor.gruppo_miscelazione_dallas_piano_terra
    internal: true  # Don't re-expose to HA

climate:
  - platform: pid
    id: fancoil_zona_1
    sensor: piano_terra_temperature  # Reference HA sensor
    # ... rest of config
```

### Completion Notes

**Implementation Summary:**
- Removed `modbus_slave`, `modbus_slave_registers`, and all `room_sensors` packages
- Removed `sensor_failover` package (three-tier Modbus failover logic)
- Replaced Modbus-based room sensors with direct Home Assistant sensor subscriptions
- Updated all 4 zone PIDs to use HA sensors: `ha_soggiorno_temperature`, `ha_cucina_temperature`, `ha_bagno_temperature`, `ha_anticamera_temperature`
- Removed all Modbus substitutions (`use_modbus`, `slave_address`, `master_address`)

**Key Changes:**
The board now gets temperature data directly from Home Assistant instead of:
1. Reading from master Modbus registers (board-to-board coordination) 
2. Reading from room Modbus sensors (individual RS485 room sensors)
3. Using complex sensor_failover logic

**New Architecture:**
- Each zone PID controller subscribes to a Home Assistant temperature sensor
- Climate mode coordination continues via HA text sensor (`sensor.thermostat_mode`)
- No Modbus infrastructure required
- Simpler, more reliable sensor path

**Home Assistant Sensor Requirements:**
The following HA sensors must be configured before deployment:
- `sensor.room_soggiorno_temperature`
- `sensor.room_cucina_temperature`
- `sensor.room_bagno_temperature`
- `sensor.room_anticamera_temperature`
