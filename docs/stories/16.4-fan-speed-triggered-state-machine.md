## User Story

As a **MEV control system**,  
I want **a fan-speed-triggered state machine that escalates equipment when the fan is working hard but losing the battle**,  
So that **humidity control uses MINIMUM equipment to stay within the 40-60% window**.

---

## Story Context

**Design Evolution (from Epic Brief v5):**

The final design uses **fan speed as BOTH the control variable AND the escalation signal**:
- Fan speed modulates continuously based on humidity position (40-60% window) and rate of change
- **Escalation** occurs when fan is working hard (≥70% or ≥80%) but humidity is still rising (rate > 0)
- **De-escalation** occurs when humidity is low (<40%) AND fan is coasting (<30%)

**Three States:**

| State             | Equipment                         | Entry Condition                            | Exit Condition                                                                        |
| ----------------- | --------------------------------- | ------------------------------------------ | ------------------------------------------------------------------------------------- |
| **FAN_ONLY**      | Fan (20-90%)                      | Default / de-escalated from DEHUMIDIFYING  | Fan ≥ 70% for 5min AND rate > 0                                                       |
| **DEHUMIDIFYING** | Fan + Dehumidifier                | Escalated from FAN_ONLY                    | Fan ≥ 70% for 5min AND rate > 0 (up) OR humidity < 40% AND fan < 30% for 10min (down) |
| **COOLING**       | Fan + Dehumidifier + Cooling Pump | Escalated from DEHUMIDIFYING (summer only) | Humidity < 40% AND fan < 30% for 10min                                                |

**Key Philosophy:**

- In Milan summer, dehumidifier and cooling are **normal operations**, not emergencies
- Fan speed reflects how hard the system is working—high fan + rising humidity = need more equipment
- Night mode is organic: as ambient humidity drops, fan speed naturally decreases → de-escalation triggers

- **HA Imports:** Imports 10+ configuration parameters from HA helpers to keep logic tunable.
- **AQ Demand:** Calculates 20-90% fan demand based on max CO₂/IAQ across 8 rooms.
- **Humidity Demand:** Calculates fan demand based on humidity window position (40-60%) + rate of change.
- **State Machine:** `select` entity `mev_humidity_state` controls relays and tracks escalation levels.
- **Transition Logic:** `interval` (1min) checks thresholds and counters for timed transitions.
- **DAC Control:** Orchestrates the 0-10V output based on final demand max(AQ, humidity, min_fan).

### Fan Speed Formula (MEV Humidity Demand)

The fan speed is calculated based on:
1. **Position in 40-60% window** (contributes 0-50% demand)
2. **Rate of change** (+20% per 1%/min when rising, capped at +30%)
3. **Minimum fan speed** (default 20%)

```yaml
template:
  - sensor:
      - name: "MEV Humidity Demand"
        unique_id: mev_humidity_demand
        unit_of_measurement: "%"
        icon: mdi:water-percent
        state: >
          {% set upper = states('input_number.mev_humidity_upper_bound') | float(60) %}
          {% set lower = states('input_number.mev_humidity_lower_bound') | float(40) %}
          {% set humidity = states('sensor.first_floor_max_humidity') | float(50) %}
          {% set rate = states('sensor.first_floor_humidity_rate') | float(0) %}
          {% set min_fan = states('input_number.mev_minimum_fan_speed') | int(20) %}
          
          {# Position in the humidity window (0% at lower, 100% at upper) #}
          {% set window_size = upper - lower %}
          {% set position = ((humidity - lower) / window_size) | float %}
          {% set position_clamped = [[position, 0] | max, 1] | min %}
          
          {# Position contributes 0-50% fan speed #}
          {% set position_factor = (position_clamped * 50) | int %}
          
          {# Rate contributes +20% per 1%/min (only when rising) #}
          {% set rate_factor = [[rate * 20, 0] | max, 30] | min | int %}
          
          {# Final calculation #}
          {{ [[min_fan + position_factor + rate_factor, 90] | min, min_fan] | max }}

      - name: "MEV Final Fan Speed"
        unique_id: mev_final_fan_speed
        unit_of_measurement: "%"
        icon: mdi:fan
        state: >
          {% set aq = states('sensor.mev_air_quality_demand') | int(20) %}
          {% set humidity = states('sensor.mev_humidity_demand') | int(20) %}
          {% set min_fan = states('input_number.mev_minimum_fan_speed') | int(20) %}
          {{ [[aq, humidity, min_fan] | max, 100] | min }}
```

### Escalation Automations

**1. FAN_ONLY → DEHUMIDIFYING:**
```yaml
automation:
  - alias: "MEV Escalate: Fan Only → Dehumidifying"
    id: mev_escalate_to_dehumidifying
    trigger:
      - platform: template
        value_template: >
          {{ states('sensor.mev_final_fan_speed') | float(0) >= 
             states('input_number.mev_escalation_fan_threshold') | float(70) and
             states('sensor.first_floor_humidity_rate') | float(0) > 0 }}
        for:
          minutes: "{{ states('input_number.mev_escalation_delay_minutes') | int(5) }}"
    condition:
      - condition: state
        entity_id: input_select.mev_humidity_state
        state: "fan_only"
    action:
      - service: input_select.select_option
        target:
          entity_id: input_select.mev_humidity_state
        data:
          option: "dehumidifying"
      - service: switch.turn_on
        target:
          entity_id: switch.first_floor_mev_dehumidifier
      - service: system_log.write
        data:
          message: "MEV: Escalated to DEHUMIDIFYING (fan ≥70%, humidity rising)"
          level: info
```

**2. DEHUMIDIFYING → COOLING:**

## Notes

- **Fan as signal:** The critical insight is that fan speed reflects how hard the system is working. When fan is maxed out but humidity is still rising, it's time to add equipment.
- **Organic night mode:** As ambient humidity drops at night, fan demand naturally decreases → triggers de-escalation. No special "night mode" logic needed.
- **Milan summer reality:** In Milan, summer is humid enough that dehumidifier and cooling are normal operations, not exceptional. This is why the design treats them as routine equipment escalation rather than emergency modes.
- **MEV intelligence:** The MEV unit has internal logic to compare inside vs outside humidity and optimize fresh/recirculated air mix. We trust the MEV to do this and focus our automation on WHEN to use each capability level.
