# Story 8.3: Room Control Coordinator Component - New Component

**Epic:** 8 - Unified State Machine Architecture  
**Story Type:** New Component Development  
**Estimated Effort:** 6 hours  
**Priority:** Critical (Core Engine)  
**Date:** October 31, 2025

---

## User Story

As a **system owner**,  
I want **a stateless coordinator component that reads condition states and coordinates PID control through priority-based decision making**,  
So that **multiple shutdown conditions (emergency, window, occupancy) are handled through a unified control engine with clear observability**.

---

## Story Context

**Existing System Integration:**

- **New component:** `components/room_control_coordinator.yaml` (no prior equivalent)
- **Replaces control logic from:** 
  - `components/room_emergency_shutdown.yaml` (Epic 5)
  - PID control logic in `components/room_window_detection.yaml` (Epic 7)
- **Technology:** ESPHome YAML, C++ lambdas, ESPHome Climate API
- **Follows pattern:** Stateless polling pattern similar to existing interval-based checks
- **Touch points:**
  - Reads global variables from condition components
  - Controls PID climate entity via `climate::ClimateCall`
  - Exposes diagnostic text sensor to HA

**Core Responsibilities:**

- Read state+priority globals from all conditions (emergency, window, future: occupancy)
- Apply priority hierarchy algorithm (lowest number = highest priority)
- Force PID to OFF when any condition Active/Recovering
- Emit diagnostic sensor showing active condition or "Normal (All Clear)"
- Log coordination decisions for debugging

## Acceptance Criteria

### Functional Requirements

1. **Condition State Reading:**
   - Reads `${zone_slug}_emergency_state` and `${zone_slug}_emergency_priority` globals
   - Reads `${zone_slug}_window_state` and `${zone_slug}_window_priority` globals
   - Polling interval: 5 seconds (via `interval:` component)
   - Validates priority values are in range 1-99 (logs warning if invalid)

2. **Priority Resolution Algorithm:**
   - Filters conditions where state > 0 (Active=1 or Recovering=2)
   - Among filtered conditions, selects one with lowest priority number
   - Stores: `highest_priority_value` (int), `active_condition_name` (string)
   - If no conditions active: `highest_priority_value = 99` (sentinel)

3. **PID Control Logic:**
   - When `highest_priority_value < 99`:
     - If PID not in OFF mode → Force PID to OFF via `climate.control`
     - If PID already OFF → No action (maintain shutdown)
     - Log: "Shutdown due to <condition>" at WARN level
   - When `highest_priority_value == 99`:
     - Log: "Normal - all conditions clear" at VERBOSE level
     - Do NOT auto-resume PID (MVP: HA automation handles resume)

4. **Diagnostic Text Sensor:**
   - Entity ID: `${zone_slug}_coordinator_status`
   - Name: `"${zone_name} Control Status"`
   - Updates every 5s
   - Displays:
     - `"Normal (All Clear)"` when all conditions state=0
     - `"Shutdown: Emergency (Active)"` when emergency highest priority and Active
     - `"Shutdown: Emergency (Recovering)"` when emergency highest priority and Recovering
     - `"Shutdown: Window (Active)"` when window highest priority and Active
     - `"Shutdown: Window (Recovering)"` when window highest priority and Recovering

5. **Edge Case Handling:**
   - **Simultaneous triggers:** Priority hierarchy resolves deterministically
   - **Invalid priority:** Log warning, ignore that condition, process remaining
   - **Missing globals:** Graceful handling (assume Normal if global undefined)

### Integration Requirements

6. **Component Interface:**
   - Accepts required vars:
     - `zone_slug`: Room identifier
     - `zone_name`: Display name for logging
     - `pid_id`: Climate entity ID to control
   - No optional vars in MVP (future: condition list, timeout overrides)

7. **Coordinator Design Compliance:**
   - Implements algorithm from `epic-8-coordinator-design.md`
   - Stateless operation (all state in condition components)
   - No direct sensor monitoring (reads condition states only)

8. **Backward Compatibility:**
   - Can coexist with Epic 5/7 components during phased migration
   - Uses unique entity IDs (no conflicts)
   - Does not interfere with existing PID control from Epic 5/7

### Quality Requirements

9. **Code Quality:**
   - Header comment documents purpose, design doc reference, required vars
   - C++ lambda follows ESPHome climate API best practices
   - Clear comments explaining priority resolution logic
   - Logging at appropriate levels (WARN for shutdown, VERBOSE for normal)

10. **Testing Validation:**
    - Manual test: Set emergency_state=1 → verify PID forced OFF
    - Manual test: Set window_state=1 → verify PID forced OFF
    - Manual test: Set both states=1 → verify emergency shown (priority 1 < 2)
    - Manual test: Clear emergency (state=0), window still 1 → verify switches to window
    - Manual test: All states=0 → verify "Normal (All Clear)" displayed

11. **Documentation:**
    - Component header references `epic-8-coordinator-design.md v1.0`
    - Inline comments for priority algorithm
    - Example usage in migration docs

## Technical Notes

**Implementation Approach:**

1. Create new file `components/room_control_coordinator.yaml`
2. Define defaults section with `condition_id` list (for future extensibility)
3. Implement 5-second interval with lambda containing:
   - Struct for condition state storage
   - Loop through known conditions (emergency, window)
   - Priority resolution algorithm
   - PID control decision tree
4. Implement text_sensor with lambda showing coordinator status
5. Add logging for all coordination decisions

**Pseudocode (from design doc):**

```cpp
interval:
  - interval: 5s
    then:
      - lambda: |-
          // Read condition states
          int emergency_state = id(${zone_slug}_emergency_state);
          int emergency_priority = id(${zone_slug}_emergency_priority);
          int window_state = id(${zone_slug}_window_state);
          int window_priority = id(${zone_slug}_window_priority);
          
          // Priority resolution
          int highest_priority = 99;
          std::string active_condition = "None";
          
          if (emergency_state > 0 && emergency_priority < highest_priority) {
            highest_priority = emergency_priority;
            active_condition = "Emergency";
          }
          
          if (window_state > 0 && window_priority < highest_priority) {
            highest_priority = window_priority;
            active_condition = "Window";
          }
          
          // PID control
          auto pid = id(${pid_id});
          if (highest_priority < 99) {
            if (pid->mode != climate::CLIMATE_MODE_OFF) {
              ESP_LOGW("coordinator", "${zone_name}: Shutdown due to %s", active_condition.c_str());
              auto call = pid->make_call();
              call.set_mode(climate::CLIMATE_MODE_OFF);
              call.perform();
            }
          } else {
            ESP_LOGV("coordinator", "${zone_name}: All conditions clear");
          }
```

**Existing Pattern Reference:**

- Interval polling: `components/room_emergency_shutdown.yaml` (lines 30-50)
- Climate API usage: `components/room_window_detection.yaml` (lines 80-95)
- Text sensor with lambda: `components/room_sensors.yaml` (lines 90-110)

**Key Constraints:**

- Must NOT auto-resume PID (MVP: HA automation handles resume)
- Must handle missing globals gracefully (defensive coding)
- Must log all coordination decisions for debugging

- Test with real conditions (Story 8.4) before production rollout

- [ ] No conflicts with Epic 5/7 during coexistence period
- [ ] Climate API usage follows ESPHome best practices
- [ ] Performance impact negligible (5s polling, <10ms CPU per cycle)

## Testing Scenarios

### Scenario 1: Single Condition Active

**Setup:** emergency_state=1, window_state=0  
**Expected:** PID OFF, status="Shutdown: Emergency (Active)"

### Scenario 2: Multiple Conditions Active

**Setup:** emergency_state=1, window_state=1  
**Expected:** PID OFF, status="Shutdown: Emergency (Active)" (priority 1 < 2)

### Scenario 3: Priority Switch During Operation

**Setup:** 
- T=0s: emergency_state=1, window_state=1 (both active)
- T=30s: emergency_state=0 (cleared), window_state=1 (still active)

**Expected:**
- T=0s: status="Shutdown: Emergency (Active)"
- T=30s: status="Shutdown: Window (Active)" (priority switch logged)

### Scenario 4: Recovery State

**Setup:** emergency_state=2 (recovering), window_state=0  
**Expected:** PID OFF, status="Shutdown: Emergency (Recovering)"

### Scenario 5: All Conditions Normal

**Setup:** emergency_state=0, window_state=0  
**Expected:** PID not changed by coordinator, status="Normal (All Clear)"
