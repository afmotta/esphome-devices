# Story 1.3: Modbus Slave - Master Data Reading and Response

## Status
Ready for Development

## Story
**As a** slave distribution board (distribuzione-piano-terra),  
**I want** to read coordination data via Modbus from the master controller,  
**so that** I can operate independently without Home Assistant sensor dependencies.

## Acceptance Criteria

1. Create reusable component `components/modbus_slave.yaml` that:
   - Implements slave device functionality (responds to master only)
   - Reads Modbus holding registers from master (address 1)
   - Exposes read values as ESPHome sensors
   - Supports parameterization via `vars:` (master address, register map, update interval)
   - Implements timeout handling and error sensors
   - Can expose local slave status registers for master to poll
2. `distribuzione-piano-terra` (slave) reads from master via Modbus:
   - Reads register 200 from master → `modbus_climate_mode` (heat/cool/off)
   - Reads register 300 from master → `modbus_master_heartbeat`
3. Slave attempts to read master registers every 10 seconds when Modbus enabled
4. Timeout handling: If Modbus read fails for >30 seconds, sensor reports "unavailable"
5. Diagnostic sensor shows Modbus read error count and last error timestamp
6. Slave exposes own status registers for master polling (e.g., register 1000: slave health status)
7. Room sensor integration framework in place (temperature/humidity reading support), with technology selection deferred to Story 1.7

### Slave Architecture Pattern

**Read-Only Consumer:**
Slaves in this architecture are **passive consumers** of master data:
- Do NOT initiate communication
- Respond only when master polls them (for status registers)
- Read master registers at fixed intervals (10s)
- No slave-to-slave communication

**Why This Pattern:**
- Simplifies RS485 bus arbitration (only master initiates)
- Prevents bus collisions and communication errors
- Standard industrial Modbus practice
- Easier to debug and maintain

[Source: architecture.md#6.1-new-component-packages-modbus-slave]

### Register Reading Implementation

**ESPHome Modbus Sensor Platform:**
```yaml
sensor:
  - platform: modbus_controller
    modbus_controller_id: modbus_slave
    id: modbus_climate_mode
    name: "Modbus Climate Mode"  # Exposed to HA for monitoring
    address: 0x00C8  # Register 200
    register_type: holding
    value_type: U_WORD
    update_interval: 10s
    filters:
      - timeout: 30s  # Mark unavailable after 3 missed reads
    internal: false  # Expose to HA for visibility
```

**Key Configuration Points:**
- `platform: modbus_controller` — Uses modbus_controller from Story 1.1
- `modbus_controller_id` — References slave controller ID
- `address` — Register address on **master** (not slave's own address)
- `register_type: holding` — Reading holding registers (function code 0x03)
- `value_type: U_WORD` — Unsigned 16-bit word
- `update_interval` — How often to poll master (10s default)
- `filters: timeout` — Mark sensor unavailable after missed reads
- `internal: false` — Expose to HA for monitoring during this story

[Source: architecture.md#6.1-new-component-packages-modbus-slave]

### Timeout Handling and Error Detection

**Three-Tier Timeout Strategy:**
1. **Read Timeout (10s):** Individual Modbus read operation timeout
2. **Sensor Timeout (30s):** Mark sensor unavailable after 3 consecutive failures
3. **Emergency Timeout (5 min):** Trigger safe shutdown (implemented in Story 1.4)

**Error Tracking Implementation:**
```yaml
globals:
  - id: modbus_read_error_count
    type: int
    restore_value: false
    initial_value: '0'

sensor:
  - platform: template
    name: "Modbus Read Error Count"
    id: modbus_error_count_sensor
    lambda: 'return id(modbus_read_error_count);'
    update_interval: 10s

text_sensor:
  - platform: template
    name: "Last Modbus Error"
    id: last_modbus_error
    # Updated by lambda when read fails
```

**On Read Failure:**
```yaml
on_value:
  - if:
      condition:
        lambda: 'return !id(modbus_climate_mode).has_state();'
      then:
        - lambda: 'id(modbus_read_error_count) += 1;'
        - text_sensor.template.publish:
            id: last_modbus_error
            state: !lambda 'return "Read failed at " + id(sntp_time).now().strftime("%Y-%m-%d %H:%M:%S");'
```

### Slave Status Register Design

**Purpose:** Allow master to monitor slave health without relying on Home Assistant.

**Register 1000 (0x03E8) - Slave Health Status:**
- Value 0: Offline/Not Ready
- Value 1: Online and Healthy
- Value 2: Degraded (errors present but operational)
- Value 3: Emergency Mode (failover active)

**Implementation:**
```yaml
number:
  - platform: modbus_controller
    modbus_controller_id: modbus_slave
    id: slave_health_status
    address: 0x03E8  # Register 1000
    value_type: U_WORD
    internal: true
    min_value: 0
    max_value: 3

# Update health status based on operational state
interval:
  - interval: 10s
    then:
      - lambda: |-
          int status = 1;  // Default: healthy
          if (id(modbus_read_error_count) > 0) {
            status = 2;  // Degraded
          }
          if (!id(modbus_climate_mode).has_state()) {
            status = 0;  // Offline
          }
          id(slave_health_status).publish_state(status);
```

**Master Polling (Future):**
Master can periodically read this register to detect slave issues and alert Home Assistant.

[Source: architecture.md#5.1-modbus-register-map]

### Component Parameterization Pattern

**Vars Contract for Reusability:**
```yaml
# In components/modbus_slave.yaml header
# Component: modbus_slave.yaml
# Purpose: Read master Modbus registers with timeout handling
# Required vars:
#   - master_address: Modbus address of master (e.g., 0x01)
#   - slave_address: This slave's Modbus address (e.g., 0x02)
# Optional vars:
#   - update_interval: Polling interval (default: 10s)
#   - timeout: Read timeout (default: 30s)

defaults:
  vars:
    master_address: 0x01
    slave_address: 0x02
    update_interval: 10s
    timeout: 30s
```

**Usage in Device YAML:**
```yaml
packages:
  modbus_slave: !include
    file: ../components/modbus_slave.yaml
    vars:
      master_address: 0x01
      slave_address: 0x02
```

This enables multiple slave boards with different addresses to use the same component.

#### Integration Test Scenarios

**Test 1: Climate Mode Synchronization**
1. On master: Change climate mode in Home Assistant (heat → cool)
2. Master log: Register 200 updates from 1 → 2
3. Slave log: Read register 200, receive value 2
4. Verify slave Modbus sensor updates within 10 seconds

**Test 2: Heartbeat Monitoring**
1. Enable Modbus on both master and slave
2. Monitor slave's modbus_master_heartbeat sensor

3. Introduce errors: Master reads value 2 (degraded)
