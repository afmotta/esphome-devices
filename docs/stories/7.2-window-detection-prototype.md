- No entity ID conflicts or naming collisions

**FR3:** OTA deployment to ground floor device succeeds:
- Firmware uploaded via `esphome upload` or HA ESPHome dashboard
- Device comes online after update
- All existing entities remain available in HA
- No climate control disruption during deployment

### Integration Requirements (Hardware Validation)

**IR1:** Window state properly integrated from Home Assistant:
- Binary sensor `binary_sensor.soggiorno_window` available in ESPHome logs
- Window open/close events detected within 10 seconds
- If HA sensor doesn't exist, create mock sensor for testing

**IR2:** State machine transitions validated on real hardware:
- Open window for 2 minutes → State remains "Normal" (grace period working)
- Keep window open for 4 minutes → State transitions to "Window Open" at 3 minutes
- Close window → State transitions to "Window Recovering"
- Wait 3 minutes with window closed → State returns to "Normal"
- All transitions logged and observable

**IR3:** PID climate control responds correctly:
- In "Normal" state: PID operates normally (heating/cooling as configured)
- Transition to "Window Open": PID mode changes to OFF within 5 seconds
- In "Window Open" state: PID remains OFF, output at 0%
- Transition to "Recovering": PID remains OFF during 3-minute stability period
- Return to "Normal": PID resumes previous mode (heating or cooling)

**IR4:** Mode-aware shutdown validated:
- Test in cooling mode with window open > 3 min → Shutdown occurs
- Test in heating mode with window open > 3 min → Shutdown occurs
- (Both modes in `shutdown_modes` list, both should trigger shutdown)

**IR5:** Diagnostic visibility in Home Assistant:
- Text sensor `text_sensor.soggiorno_window_state` appears in HA
- State updates in real-time as transitions occur
- State values are human-readable: "Normal", "Window Open", "Window Recovering"
- Can be added to HA dashboard for monitoring

### Quality Requirements

**QR1:** Existing ground floor functionality regression tested:
- All 4 ground floor rooms maintain temperature control
- Non-soggiorno rooms (cucina, bagno, anticamera) unaffected by update
- Slow PWM outputs, relay control patterns unchanged
- No increase in errors or warnings in ESPHome logs

**QR2:** Graceful degradation tested:
- Simulate window sensor unavailable (disconnect in HA or delete entity temporarily)
- Verify soggiorno climate control continues normally
- Verify state remains "Normal" (sensor treated as closed)
- Verify no errors logged related to missing sensor

**QR3:** Edge cases validated:
- Window opened and closed repeatedly < 3 minutes → No shutdown (grace period resets)
- Window opened while PID already OFF (manual override) → State tracks correctly
- Multiple simultaneous shutdowns (window + sensor emergency) → Both work without conflict
- Mode changes while window open → Shutdown respects new mode's inclusion in `shutdown_modes`

## Testing Scenarios (Real Hardware)

### Scenario 1: Normal Operation Baseline
- **Given:** Fresh deployment, window closed
- **When:** Monitor for 10 minutes
- **Then:** State remains "Normal", PID operates normally, temperature controlled

### Scenario 2: Brief Window Opening (< Timeout)
- **Given:** System in "Normal" state
- **When:** Open window for 2 minutes, then close
- **Then:** State remains "Normal" throughout, no shutdown, no PID interruption

### Scenario 3: Extended Window Opening (> Timeout)
- **Given:** System in "Normal" state
- **When:** Open window and leave open for 5 minutes
- **Then:** 
  - At 0-3 min: State remains "Normal", PID active
  - At 3 min: State transitions to "Window Open", PID turns OFF
  - At 3-5 min: State remains "Window Open", PID remains OFF

### Scenario 4: Window Close and Recovery
- **Given:** System in "Window Open" state (window open for >3 min)
- **When:** Close window
- **Then:**
  - Immediately: State transitions to "Window Recovering"
  - 0-3 min recovering: State remains "Window Recovering", PID remains OFF
  - After 3 min: State returns to "Normal", PID resumes operation

### Scenario 5: Rapid Open/Close Cycles
- **Given:** System in "Normal" state
- **When:** Open window for 1 min, close for 30 sec, open for 1 min, close (repeat 3 times)
- **Then:** State remains "Normal" throughout (grace period never exceeded), no shutdowns

### Scenario 6: Window Sensor Unavailable
- **Given:** System operating normally
- **When:** Make window sensor unavailable in HA (disconnect, delete, or set to "unavailable")
- **Then:** State remains "Normal", climate control continues, no errors logged

### Scenario 7: Mode Change While Window Open
- **Given:** System in cooling mode, window open for 5 min (shutdown triggered)
- **When:** Change PID mode to heating via HA
- **Then:** System respects heating mode in `shutdown_modes`, keeps PID OFF (or resumes depending on configuration)

### Scenario 8: Manual PID Override
- **Given:** System in "Normal" state with window closed
- **When:** Manually turn OFF PID via HA, then open window for 5 minutes
- **Then:** State transitions to "Window Open", PID remains OFF, no conflicts with manual state

## Risk Assessment

### Primary Risks

**Risk 1: Window sensor doesn't exist in HA**
- **Impact:** Cannot test real window detection
- **Mitigation:** Create mock binary sensor in HA for testing (input_boolean)
- **Validation:** Document actual sensor entity IDs needed for production

**Risk 2: Firmware size exceeds flash limits with window detection**
- **Impact:** Cannot deploy to device
- **Mitigation:** Check firmware size during compilation, optimize if needed
- **Rollback:** Revert soggiorno configuration, redeploy previous firmware

**Risk 3: State machine causes PID to become unresponsive**
- **Impact:** Soggiorno climate control fails
- **Mitigation:** Monitor ESPHome logs during testing, identify conflicts
- **Rollback:** Remove window detection package, redeploy

**Risk 4: Timing inaccuracies in state transitions**
- **Impact:** Shutdown occurs too early or late, user experience degraded
- **Mitigation:** Test with stopwatch, adjust timeouts if needed
- **Resolution:** Document actual timing behavior, adjust component if necessary

### Rollback Plan

**If prototype reveals critical issues:**
1. Remove window detection package from `soggiorno.yaml`
2. Recompile and OTA update ground floor device
3. Soggiorno returns to Epic 4 behavior (no window detection)
4. Document issues discovered for component refinement
5. Fix component (Story 7.1), retry Story 7.2
