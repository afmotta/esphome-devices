# Story 2.1: Mixing Valve PID Simplification (gruppo-miscelazione)

## Status
In Progress

## Story
**As a** mixing valve controller (gruppo-miscelazione),  
**I want** to use a single PID climate entity per circuit instead of dual heat/cool entities,  
**so that** the architecture is simpler, easier to maintain, and aligned with standard ESPHome patterns.

## Acceptance Criteria

1. Replace `mixing_valve.yaml` component usage with direct single PID climate configuration
2. Each mixing valve circuit has ONE climate entity instead of two:
   - Old: `climate.pid_piano_terra_heat` + `climate.pid_piano_terra_cool`
   - New: `climate.pid_piano_terra` (supports both heat and cool modes)
3. Remove dependency on `dual_pid.yaml` and `valve_trigger.yaml` components
4. Maintain all existing PID tuning parameters (Kp, Ki, Kd) for both modes
5. Mode coordination based on global `climate_mode` sensor (inherited from Modbus master)
6. Test in local environment:
   - Configuration validates without errors
   - Firmware compiles successfully
   - Both mixing valves control correctly in heat and cool modes
7. Document entity ID mapping for Home Assistant migration

## Current Architecture Analysis

**Current Setup (gruppo-miscelazione.yaml):**
```yaml
packages:
  mixing_valve_terra: !include
    file: ../components/mixing_valve.yaml
    vars:
      circuit_name: "Piano Terra"
      circuit_slug: piano_terra
      sensor: dallas_0x81000000b3e6f628
      switch: relay_1
      output: dac_1
      heat_default_temperature: 28.0
      heat_kp: 0.2, heat_ki: 0.01, heat_kd: 0.0
      cool_default_temperature: 22.0
      cool_kp: 0.2, cool_ki: 0.01, cool_kd: 0.0
      trigger: input_1
```

**Component Chain:**
- `mixing_valve.yaml` includes:
  - `dual_pid.yaml` (creates 2 climate entities)
    - `pid.yaml` (heat mode)
    - `pid.yaml` (cool mode)
  - `valve_trigger.yaml` (heat trigger)
  - `valve_trigger.yaml` (cool trigger)

**Problems:**
1. Two climate entities per circuit when one suffices
2. Complex coordination via `valve_trigger.yaml` binary sensors
3. Mutual exclusion logic in `pid.yaml` to turn off opposite mode
4. Both PIDs control same output (`dac_1` or `dac_2`)

## Proposed Architecture

**Single PID Climate per Circuit:**
```yaml
climate:
  - platform: pid
    id: pid_piano_terra
    name: "PID Piano Terra"
    sensor: dallas_0x81000000b3e6f628
    default_target_temperature: 28.0
    heat_output: dac_1
    cool_output: dac_1
    control_parameters:
      kp: 0.2
      ki: 0.01
      kd: 0.0
    # Mode coordination via automation
```

**Mode Coordination:**
```yaml
# Simple automation based on climate_mode
on_...:
  - climate.control:
      id: pid_piano_terra
      mode: !lambda |-
        if (id(climate_mode).state == "HEAT") return CLIMATE_MODE_HEAT;
        if (id(climate_mode).state == "COOL") return CLIMATE_MODE_COOL;
        return CLIMATE_MODE_OFF;
```

**Benefits:**
- Single entity: `climate.pid_piano_terra` instead of 2
- Standard ESPHome PID (no custom components)
- Simpler logic (no mutual exclusion)
- Same output for both modes (native ESPHome behavior)

## Risk Assessment

### Risks
1. **Breaking HA automations**: Entity ID changes require HA updates
2. **PID tuning differences**: Single PID might behave differently than dual setup
3. **Mode switching logic**: Need to verify coordination with global climate_mode
4. **Relay control**: Must ensure relays turn off when PID is idle

### Mitigation
1. **Document all entity ID changes** before deployment
2. **Maintain exact PID parameters** from old config
3. **Test mode switching** extensively in local environment
4. **Add explicit relay control** automation with failsafe
