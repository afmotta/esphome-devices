# Story 3.3: Remove Modbus Slave from distribuzione-primo-piano

## Status
Ready for Review

## Story
**As a** system architect,  
**I want** to remove Modbus slave functionality from distribuzione-primo-piano and replace remote sensor reading with direct Home Assistant sensor integration,  
**so that** the distribution board operates independently without requiring board-to-board Modbus communication.

## Acceptance Criteria

1. Remove `modbus_slave` package from `devices/distribuzione-primo-piano.yaml`
2. Remove `modbus_slave_registers` package from `devices/distribuzione-primo-piano.yaml`
3. Replace `sensor_failover` logic with direct Home Assistant sensor subscriptions
4. Update fancoil configurations to use HA temperature sensors instead of Modbus failover
5. Verify all fancoils continue to operate correctly with HA-provided temperatures
6. Configuration validates and compiles without errors
7. No regression in fancoil temperature control or climate mode handling

### Architecture Context
**Current Role**: distribuzione-primo-piano is a Modbus slave board (A16 Ethernet) that:
- Reads Dallas temperature sensors from gruppo-miscelazione via Modbus registers
- Uses sensor_failover to handle Modbus disconnections with local fallback sensors
- Controls 8 fancoil zones on the first floor (primo piano)

**After Epic 3.3**: distribuzione-primo-piano becomes independent board that:
- Subscribes to Dallas sensors from gruppo-miscelazione via Home Assistant
- No Modbus slave functionality
- Fancoils use HA-provided temperature sensors directly
- Home Assistant coordinates climate modes and temperatures

### Important Notes
- Epic 2 already simplified fancoil PID architecture
- Each fancoil currently has a `sensor_failover` providing temperature to its PID
- Sensor failover reads from Modbus (primary) with local Dallas (fallback)
- Need to replace failover with direct HA sensor subscription
- This is identical pattern to Story 3.2, but for first floor instead of ground floor

### Key Preservation Points
1. **Fancoil PIDs**: Keep PID configurations, only change sensor source
2. **Relay/Output Control**: No changes to fancoil relay/valve outputs
3. **ESPHome API**: Ensure enabled for bidirectional HA integration
4. **Sensor IDs**: Update to reference HA sensor entity IDs

### Home Assistant Sensor Integration Pattern
```yaml
sensor:
  - platform: homeassistant
    id: primo_piano_temperature
    entity_id: sensor.gruppo_miscelazione_dallas_primo_piano
    internal: true  # Don't re-expose to HA

climate:
  - platform: pid
    id: fancoil_zona_1
    sensor: primo_piano_temperature  # Reference HA sensor
    # ... rest of config
```

### Completion Notes

**Implementation Summary:**
- Removed `modbus_slave`, `modbus_slave_registers`, and `sensor_failover` packages
- Replaced Modbus-based temperature failover with direct Home Assistant sensor subscription
- Updated both zone PIDs to use HA sensor: `ha_primo_piano_temperature`
- Removed all Modbus substitutions (`use_modbus`, `slave_address`, `master_address`)

**Key Changes:**
The board now gets temperature data directly from Home Assistant instead of:
1. Reading from master Modbus registers (board-to-board coordination)
2. Using sensor_failover logic (Modbus primary → HA fallback)

**New Architecture:**
- Both zone PIDs (Zona 1 and Zona 2) subscribe to single HA temperature sensor
- Sensor entity: `sensor.gruppo_miscelazione_dallas_primo_piano`
- Climate mode coordination continues via HA text sensor (`sensor.thermostat_mode`)
- No Modbus infrastructure required

**Zones Configuration:**
- **Zona 1**: Uses `ha_primo_piano_temperature` → `slow_pwm_zona_1` → `relay_01`
- **Zona 2**: Uses `ha_primo_piano_temperature` → `slow_pwm_zona_2` → `relay_02`

**Home Assistant Sensor Requirements:**
The following HA sensor must be configured before deployment:
- `sensor.gruppo_miscelazione_dallas_primo_piano` (exposed from gruppo-miscelazione Dallas sensor in Story 3.1)
