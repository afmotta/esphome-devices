# Story 5.2: Emergency Shutdown Logic Implementation

## Status
Ready for Review

## Story
**As a** room climate controller,  
**I want** automatic emergency shutdown of relays and PIDs when HA is unavailable for >3 minutes,  
**so that** the HVAC system fails safely without valid sensor data, preventing potential damage or excessive energy use.

## Acceptance Criteria

1. When room sensor enters Emergency state (from Story 5.1), trigger emergency shutdown actions
2. Emergency actions per room:
   - Turn OFF associated relay (physical output)
   - Set PID mode to OFF (climate entity shows OFF state)
   - Log ERROR message with timestamp
3. Emergency logic integrated into room component packages (not separate component)
4. Emergency actions are idempotent (safe to trigger multiple times)
5. Recovery logic implemented:
   - When sensor exits Emergency (enters Recovering), wait for stability
   - After recovery_stability_timeout (60s), restore PID to previous mode
   - Relay control restored automatically by PID operation
6. Test emergency on all three device types:
   - `gruppo-miscelazione` (mixing valve circuits with DAC outputs)
   - `distribuzione-piano-terra` (ground floor rooms with relay outputs)
   - `distribuzione-primo-piano` (first floor rooms with relay outputs)
7. Emergency trigger verified with physical hardware (relay clicks OFF)
8. Recovery verified with physical hardware (relay clicks back ON when PID resumes)
9. No false emergencies during normal operation (7-day monitoring)

### Emergency Actions (Per Room)

**Option A: Lambda-based actions in room component**
```yaml
# In room component (e.g., bagno_grande.yaml)
sensor:
  - platform: template
    id: bagno_grande_emergency_monitor
    update_interval: 5s
    lambda: |-
      if (id(bagno_grande_emergency_triggered)) {
        // Emergency actions
        auto relay_call = id(relay_12).turn_off();
        relay_call.perform();
        
        auto pid_call = id(pid_radiant_bagno_grande).make_call();
        pid_call.set_mode(CLIMATE_MODE_OFF);
        pid_call.perform();
        
        ESP_LOGE("emergency", "Bagno Grande: EMERGENCY SHUTDOWN ACTIVE");
      }
      return 0;  // Dummy return
```

**Option B: Automation-based actions**
```yaml
# In room component
on_...:  # Triggered by state change
  - if:
      condition:
        lambda: return id(${zone_slug}_emergency_triggered);
      then:
        - switch.turn_off: ${relay_id}
        - climate.control:
            id: pid_radiant_${zone_slug}
            mode: OFF
        - logger.log:
            format: "${zone_name}: EMERGENCY SHUTDOWN"
            level: ERROR
```

**Chosen Approach:** Option B (automation-based) - cleaner, more ESPHome-idiomatic

## Proposed Implementation Approach

### Add to `room_sensors.yaml` (Story 5.1)

**Option: Expose emergency state as binary sensor**
```yaml
binary_sensor:
  - platform: template
    id: ${zone_slug}_emergency_active
    lambda: return id(${zone_slug}_emergency_triggered);
```

### Add to each room component package

**In `components/rooms/*/room_name.yaml`:**
```yaml
# Existing content...
packages:
  room_sensors: !include ...
  pid_radiant: !include ...

output:
  - platform: slow_pwm
    id: slow_pwm_${zone_slug}
    turn_on_action:
      - switch.turn_on: ${relay_switch_id}
    turn_off_action:
      - switch.turn_off: ${relay_switch_id}

# NEW: Emergency shutdown logic
interval:
  - interval: 5s
    then:
      - lambda: |-
          if (id(${zone_slug}_emergency_triggered)) {
            // Emergency mode: ensure relay OFF, PID OFF
            if (id(${relay_switch_id}).state) {
              ESP_LOGW("emergency", "${zone_name}: Forcing relay OFF");
              id(${relay_switch_id}).turn_off();
            }
            
            if (id(pid_radiant_${zone_slug}).mode != CLIMATE_MODE_OFF) {
              ESP_LOGW("emergency", "${zone_name}: Forcing PID OFF");
              auto call = id(pid_radiant_${zone_slug}).make_call();
              call.set_mode(CLIMATE_MODE_OFF);
              call.perform();
            }
          }
```

**Problem:** This couples emergency logic with room component. Better to integrate into `room_sensors.yaml`?

### Alternative: Add emergency actions to `room_sensors.yaml`

**Add to `room_sensors.yaml` (requires relay_switch_id and pid_id vars):**
```yaml
# Required additional vars:
#   - relay_switch_id: ID of relay switch to control (e.g., "relay_12")
#   - pid_id: ID of PID climate entity (e.g., "pid_radiant_bagno_grande")

interval:
  - interval: 5s
    then:
      - lambda: |-
          if (id(${zone_slug}_emergency_triggered)) {
            // Emergency shutdown
            if (id(${relay_switch_id}).state) {
              id(${relay_switch_id}).turn_off();
              ESP_LOGE("emergency", "${zone_name}: Relay OFF");
            }
            
            if (id(${pid_id}).mode != CLIMATE_MODE_OFF) {
              auto call = id(${pid_id}).make_call();
              call.set_mode(CLIMATE_MODE_OFF);
              call.perform();
              ESP_LOGE("emergency", "${zone_name}: PID OFF");
            }
          } else {
            // Recovery: PID will auto-resume based on HA state
            // Relay will be controlled by PID via slow_pwm output
          }
```

**This approach:**
- ✅ Centralizes emergency logic in one place
- ✅ Reusable across all rooms
- ❌ Requires additional vars (relay_switch_id, pid_id)
- ❌ Couples sensor component with actuator control (less clean separation)

## Recommended Implementation

**Use room component level emergency logic:**

1. `room_sensors.yaml` (Story 5.1) provides:
   - Emergency state detection
   - Emergency flag: `${zone_slug}_emergency_triggered`
   - State visibility: `text_sensor.{zone}_sensor_state`

2. Room component packages add emergency actions:
   - Monitor `${zone_slug}_emergency_triggered` flag
   - Execute relay OFF + PID OFF when flag is true
   - Log actions appropriately

**Rationale:**
- Separation of concerns: Sensors detect, actuators act
- Room component knows its own relay/PID IDs
- Easier to customize per-room if needed
- Cleaner var structure (sensor component doesn't need actuator IDs)

## Testing Scenarios

### Scenario 1: Single Room Emergency
- **Given:** One room sensor unavailable
- **When:** 180 seconds elapse
- **Then:** That room enters emergency, others unaffected

### Scenario 2: System-wide HA Outage
- **Given:** HA completely down (all sensors unavailable)
- **When:** 180 seconds elapse
- **Then:** All rooms enter emergency simultaneously

### Scenario 3: Flaky Network Recovery
- **Given:** Room in emergency, HA reconnects but flaky
- **When:** HA connects/disconnects repeatedly
- **Then:** Room stays in "Recovering" until 60s continuous stability

### Scenario 4: Normal HA Restart
- **Given:** HA restarted (typical 30-60s downtime)
- **When:** HA comes back online
- **Then:** No emergency triggered (restart faster than 180s timeout)

### Scenario 5: Individual Sensor Failure
- **Given:** One HA sensor entity fails, others OK
- **When:** Affected room's sensor unavailable >180s
- **Then:** Only that room enters emergency
