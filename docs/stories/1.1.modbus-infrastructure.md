## Dev Notes

### Architecture Context

This story implements the foundational Modbus RTU infrastructure for the brownfield ESPHome climate control system. The system currently relies entirely on Home Assistant for sensor data and coordination. This enhancement adds RS485 Modbus communication as a **parallel capability** without disrupting existing functionality.

**CRITICAL PRESERVATION REQUIREMENTS:**
- All PID tuning parameters must remain untouched (months of manual tuning investment)
- Entity IDs must preserve compatibility with existing Home Assistant dashboards
- System must remain operational - this is additive only, no existing functionality removed
- RS485 UART hardware is already configured but unused - we're activating it

### ESPHome Modbus Implementation

**Required Components:**
```yaml
# UART configuration
uart:
  tx_pin: GPIO27  # Master A6, or GPIO13 for Slave A16
  rx_pin: GPIO14  # Master A6, or GPIO16 for Slave A16
  baud_rate: 9600
  stop_bits: 1
  data_bits: 8
  parity: NONE

# Modbus bus
modbus:
  id: rs485_bus
  
# Modbus controller (Master or Slave)
modbus_controller:
  - id: modbus_master  # or modbus_slave
    address: 0x1  # Master=0x1, Ground Slave=0x2, First Slave=0x3
    modbus_id: rs485_bus
    role: master  # or slave
```

**Master Capabilities:**
- Can initiate read/write operations to any slave address
- Polls slaves at configurable interval (default: 10 seconds)
- Must handle slave timeouts gracefully

**Slave Capabilities:**
- Responds only to master requests on its address
- Does not initiate communication
- Can expose holding registers for master to read

### Diagnostic Sensors Design

**Modbus Health Sensors (expose to Home Assistant):**
```yaml
sensor:
  - platform: template
    name: "Modbus Master Status"
    id: modbus_master_status
    lambda: |-
      return id(modbus_master).get_state() == modbus_controller::READY ? 1 : 0;
    update_interval: 10s

  - platform: template
    name: "Modbus Communication Errors"
    id: modbus_error_count
    # Increments on each failed poll

  - platform: template
    name: "Last Successful Modbus Poll"
    id: modbus_last_poll
    device_class: timestamp
```

These sensors allow Home Assistant monitoring of Modbus health without affecting control logic.

### ESPHome Package Composition Pattern

The project uses ESPHome's package composition for modularity:

```yaml
# Example from existing mixing_valve.yaml
packages:
  mixing_valve_piano_terra: !include
    file: ../components/mixing_valve.yaml
    vars:
      circuit_name: "Piano Terra"
      circuit_slug: "piano_terra"
      sensor: "dallas_0x81000000b3e6f628"
      switch: "relay_1"
      output: "dac_1"
```

**Key Patterns:**
- Template-Based Naming: Component IDs use `${circuit_slug}`, `${mode}` interpolation
- Multi-Level Composition: Components can include other components
- Variable substitution via `vars:` block
- Dallas Sensor Addressing: By ROM address (e.g., `0x81000000b3e6f628`)

**For This Story:**
New Modbus components should follow this pattern, accepting configuration via `vars:` for reusability.

[Source: architecture.md#2.3-software-architecture-patterns]

### Feature Flag Design

**CRITICAL:** Implement feature flag for gradual rollout and instant rollback:

```yaml
substitutions:
  use_modbus: "false"  # Feature flag: false initially for safety
```

This allows:
- A/B testing (Modbus vs. HA side-by-side)
- Instant rollback without firmware reflash (just change flag and OTA update)
- Gradual rollout per device (master → ground → first floor)

The Modbus components should check this flag and be inactive when `use_modbus: false`.

[Source: architecture.md#3.2-integration-approach]

### Coding Standards

**Entity Naming Conventions:**
- **IDs (internal):** `snake_case` with descriptive prefixes (e.g., `modbus_master_status`)
- **Names (Home Assistant):** Title Case with spaces (e.g., "Modbus Master Status")
- **Internal entities:** Use `internal: true` for implementation details not needed in HA

**Template Sensors:**
- Always include `device_class`, `state_class`, `unit_of_measurement` for proper HA integration
- Use `update_interval` matching existing patterns (10s for fast sensors, 60s for slow)

**Modbus-Specific Standards:**
- Use hexadecimal notation for register addresses: `0x0001`, not `1`
- Document register map in component header comments
- Implement `update_interval` to detect stale data
- Add timeout filters: `filters: - timeout: 30s` to mark unavailable after missed polls

**Component Structure:**
```yaml
# Component header comment with purpose and vars contract
# Component: modbus_master.yaml
# Purpose: Expose coordination data via Modbus registers
# Required vars: (list any parameterization)

# Modbus controller configuration
modbus_controller:
  # ...

# Sensors, switches, outputs, etc.
```
