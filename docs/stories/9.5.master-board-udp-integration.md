## Story

**As a** system administrator,  
**I want** the master board (gruppo-miscelazione) to broadcast supply temperatures and climate mode via UDP,  
**so that** slave boards can receive this critical data directly without relying on Home Assistant as an intermediary.

---

## Acceptance Criteria

1. `devices/gruppo-miscelazione.yaml` integrates `udp_sender.yaml` component via packages
2. UDP sender broadcasts both Dallas supply temperature sensors (piano terra + primo piano)
3. UDP sender includes climate mode text sensor in broadcasts
4. Feature flag `use_udp: true` enables UDP broadcasting
5. Board compiles successfully with UDP sender included
6. Firmware size remains within ESP32 flash limits after UDP addition
7. UDP packets visible on network (Wireshark/tcpdump) containing supply temps
8. Existing functionality unchanged (PID controllers, mixing valves, HA integration)
9. UDP diagnostic sensors visible in HA dashboard
10. Rollback capability validated by setting `use_udp: false`

## Dev Notes

### Relevant Source Tree

**Target Device File:**
- `devices/gruppo-miscelazione.yaml` (KC868-A6 master board)

**Component Dependencies:**
- `components/udp_sender.yaml` (created in Story 9.1)
- `boards/a6.yaml` (hardware config with Ethernet/WiFi)
- `boards/wifi.yaml` (network connectivity)

**Existing Sensor IDs (from gruppo-miscelazione.yaml):**
- Dallas supply temp piano terra: `dallas_0x81000000b3e6f628`
- Dallas supply temp primo piano: `dallas_0xe9000000b366ed28`
- PID controllers: `pid_mixing_ground_floor`, `pid_mixing_first_floor`

### Climate Mode Sensor Strategy

The master board needs to broadcast the current climate mode (heating/cooling) to slave boards. Options:

**Option 1: Use existing HA sensor** (if already consumed)
```yaml
sensor:
  - platform: homeassistant
    id: master_climate_mode
    entity_id: sensor.thermostat_mode
```

**Option 2: Derive from PID states** (autonomous)
```yaml
text_sensor:
  - platform: template
    id: master_climate_mode
    name: "Master Climate Mode"
    lambda: |-
      if (id(pid_mixing_ground_floor).mode == climate::CLIMATE_MODE_HEAT) {
        return {"heating"};
      } else if (id(pid_mixing_ground_floor).mode == climate::CLIMATE_MODE_COOL) {
        return {"cooling"};
      } else {
        return {"off"};
      }
```

**Recommendation:** Use Option 2 for true autonomy (no HA dependency for climate mode).

### Integration Pattern

Add to `devices/gruppo-miscelazione.yaml`:
```yaml
packages:
  base: !include ../boards/a6.yaml
  wifi: !include ../boards/wifi.yaml
  # ... existing packages ...
  
  # UDP Sender (Epic 9)
  udp_sender: !include
    file: ../components/udp_sender.yaml
    vars:
      use_udp: true
      sensors:
        - dallas_0x81000000b3e6f628
        - dallas_0xe9000000b366ed28
      text_sensors:
        - master_climate_mode
      update_interval: 10
      broadcast_port: 6053
```

### Firmware Size Considerations

Current gruppo-miscelazione firmware includes:
- 2 PID controllers with sensor monitoring components
- Dallas temperature sensors (1-Wire)
- Ethernet + WiFi networking
- PCF8574 I2C expanders

Adding UDP sender will increase firmware by ~5-10KB (JSON serialization + UDP stack). Monitor binary size during compile:
```bash
esphome compile devices/gruppo-miscelazione.yaml | grep "RAM:"
```

### Network Configuration

- Master board has static IP or DHCP reservation (verify in `boards/wifi.yaml`)
- UDP broadcasts to `xxx.xxx.xxx.255` (broadcast address for /24 network)
- Port 6053 (ESPHome default, no conflicts expected)

### Rollback Strategy

If UDP causes issues:
1. Set `use_udp: false` in gruppo-miscelazione.yaml
2. Recompile and flash via OTA
3. System reverts to Epic 8 behavior (HA-only coordination)
4. No data loss, no manual intervention on slave boards needed

### Previous Epic Context

**Epic 8:** Unified state machine with coordinator
- Master board currently only manages mixing valves
- Slave boards consume supply temps from HA `homeassistant` platform sensors
- Epic 9 will make supply temps available via UDP, but HA sensors remain as Tier 2 fallback

### Testing Standards

#### Testing Requirements
1. **Compile Test:** Firmware builds without errors
2. **Size Test:** Binary fits in flash (<1MB available)
3. **Network Test:** UDP packets visible on network capture
4. **Regression Test:** All existing PID/mixing valve functionality works
5. **Diagnostic Test:** UDP counters visible in HA dashboard
6. **Rollback Test:** Feature flag disables UDP cleanly

#### Testing Procedure
```bash
# Compile and check size
esphome compile devices/gruppo-miscelazione.yaml

# Flash via OTA
esphome upload devices/gruppo-miscelazione.yaml --device gruppo-miscelazione.local

# Monitor logs
esphome logs devices/gruppo-miscelazione.yaml --device gruppo-miscelazione.local

# Network capture (separate machine)
sudo tcpdump -i eth0 -n udp port 6053 -A

# Validate JSON payload
tcpdump -i eth0 -n udp port 6053 -w /tmp/udp.pcap
tshark -r /tmp/udp.pcap -T fields -e data.text
```
