# Story 1.6: Ground Floor Fancoil 0-10V Control via Modbus Adapter

## Status
Draft

## Story
**As a** ground floor distribution board (distribuzione-piano-terra),  
**I want** to control 4 fancoils via a 4-channel Modbus 0-10V adapter,  
**so that** fancoil capacity can be precisely regulated based on zone temperature and climate mode.

## Acceptance Criteria

### 0-10V Adapter Configuration

1. Configure 4-channel Modbus 0-10V adapter:
   - Modbus slave address: 5 (ground floor fancoil adapter)
   - 4 independent channels for 4 fancoils:
     - Channel 1: Soggiorno Fancoil 1 (register 0x0001)
     - Channel 2: Soggiorno Fancoil 2 (register 0x0002)
     - Channel 3: Cucina Fancoil (register 0x0003)
     - Channel 4: Technical Closet Fancoil (register 0x0004)
   - Each register range: 0-10000 (0V = off, 10000 = 10V full capacity)

2. Create multi-channel 0-10V component: `components/modbus_0_10v_multi.yaml`
   - Supports multiple output channels via vars
   - Each channel has independent number platform and template output
   - Parameterization: adapter_address, channel_count, channel_names, output_ids
   - Diagnostic sensors: per-channel status, last update, error count

### Fancoil Control Logic

1. Integrate 0-10V adapter into `distribuzione-piano-terra` device:
   - Add modbus_0_10v_multi package with adapter address 5
   - Create 4 template outputs for fancoil control
   - Connect each output to corresponding adapter register

2. Implement PID-based fancoil control for each zone:
   - **Soggiorno**: Dual fancoils controlled by single PID output (split 50/50 or sequential staging)
   - **Cucina**: Single fancoil controlled by PID output
   - **Technical Closet**: Single fancoil controlled by PID output
   - Each PID controller uses room temperature sensor (Home Assistant initially)

3. Climate mode awareness:
   - Fancoils only active in cooling mode (climate_mode = 2)
   - In heating mode, fancoils disabled (0V output)
   - Smooth transition when mode changes

4. Fancoil staging logic for Soggiorno (2 fancoils):
   - **Option A - Split**: Both fancoils at 50% of PID output (0-50% PID → both at 0-50%)
   - **Option B - Sequential**: First fancoil 0-100%, second activates when first at 100%
   - User-selectable via substitution variable

### Performance and Safety

1. Response time requirements:
   - PID output change → 0-10V register update within 2 seconds
   - Mode change (heat→cool) → fancoils respond within 5 seconds

2. Safety features:
   - Emergency shutdown integration: All fancoils to 0V when emergency mode
   - Failover integration: Fancoils controlled via failover temperature sensors
   - Manual override capability via Home Assistant number entities

3. Diagnostic monitoring:
   - Per-fancoil capacity percentage exposed to HA
   - Modbus communication status for adapter
   - Total cooling output aggregation sensor

### System Integration

1. RS485 bus integration:
   - 0-10V adapter (address 5) added to existing bus
   - Ground floor A16 acts as Modbus master to adapter
   - No conflicts with other devices (master=1, slaves=2/3, second-floor-adapter=4, ground-fancoil-adapter=5)

2. Home Assistant integration:
   - 4 fancoil capacity sensors visible in HA
   - Manual control numbers for testing/override
   - Aggregate "Total Cooling Capacity" sensor
   - Per-zone cooling status binary sensors

3. Existing functionality preserved:
   - Existing relay-based fancoil control (if any) can coexist
   - PID controllers for floor heating/cooling unchanged
   - No impact on mixing valve control

## Dev Notes

### Architecture Context

This story adds **precise 0-10V control** for the 4 ground floor fancoils via a 4-channel Modbus adapter. Currently, the ground floor uses fancoils for cooling, but the existing implementation likely uses simple relay on/off control. This enhancement provides:

- **Proportional control**: Fancoil capacity adjusts smoothly based on cooling demand
- **Zone-specific control**: Each zone's fancoil responds to its room temperature
- **Staging logic**: Soggiorno's 2 fancoils can be split or sequentially staged
- **Integration with existing failover**: Fancoils use same three-tier sensor failover

**Device Layout - Ground Floor:**
- **Soggiorno** (Living Room): 2 fancoils - largest zone, highest cooling load
- **Cucina** (Kitchen): 1 fancoil - medium zone
- **Technical Closet**: 1 fancoil - small zone, equipment cooling

**CRITICAL ARCHITECTURAL DECISION - Mode-Specific Outputs:**

The ground floor uses **different physical systems** for heating vs cooling:
- **Heating Mode**: Floor radiators (existing `slow_pwm_*` outputs) - low temperature, slow response
- **Cooling Mode**: Fancoils via 0-10V adapter - high capacity, fast response

**Impact on `dual_pid.yaml` Component:**

The current `dual_pid.yaml` component assumes the **same output** is used for both heat and cool modes:
```yaml
climate:
  - id: !extend pid_${circuit_slug}_heat
    heat_output: ${output}  # Same output for both!
  - id: !extend pid_${circuit_slug}_cool
    cool_output: ${output}  # Same output for both!
```

This works for systems where the same actuator handles both modes (e.g., mixing valves), but **NOT for ground floor** where:
- Heat mode → Floor radiator output (`slow_pwm_soggiorno`)
- Cool mode → Fancoil output (`soggiorno_fc1` + `soggiorno_fc2`)

**Resolution Options:**

1. **Option A - Extend dual_pid.yaml** (Recommended):
   - Add optional vars: `heat_output` and `cool_output` (separate outputs)
   - If only `output` provided: use for both modes (backward compatible)
   - If both provided: use mode-specific outputs
   
2. **Option B - Create new component** (More explicit):
   - Create `components/split_mode_pid.yaml` for mode-specific outputs
   - Keep `dual_pid.yaml` unchanged for backward compatibility
   - Ground floor uses new component

3. **Option C - Device-level configuration** (Quick fix):
   - Keep dual_pid.yaml unchanged
   - Override climate entity outputs at device level using `!extend`
   - Less reusable but maintains component simplicity

**Recommendation**: **Option A** - Extend `dual_pid.yaml` to support optional mode-specific outputs. This maintains backward compatibility while enabling the ground floor's split heating/cooling architecture.

- This story reuses the 0-10V component pattern but adapts for multi-channel

This works for mixing valves (one actuator for both modes) but fails for ground floor where:

### Multi-Channel 0-10V Component Design

The `modbus_0_10v_multi.yaml` component extends the single-channel pattern from Story 1.5:

```yaml
# Component: modbus_0_10v_multi.yaml
# Purpose: Control multi-channel 0-10V Modbus adapter for multiple fancoils
# Vars: adapter_address, channel_count, channel_names[], output_ids[], register_base (default 1)

modbus_controller:
  - id: modbus_0_10v_multi_adapter
    address: ${adapter_address}
    modbus_id: rs485_bus
    role: master

# Loop for each channel (ESPHome supports substitution loops)
# Channel 1: Soggiorno Fancoil 1
number:
  - platform: modbus_controller
    modbus_controller_id: modbus_0_10v_multi_adapter
    id: ${output_ids[0]}_register
    name: "${friendly_name} ${channel_names[0]} Register"
    address: 0x0001
    value_type: U_WORD
    min_value: 0
    max_value: 10000
    internal: true

output:
  - platform: template
    id: ${output_ids[0]}
    type: float
    write_action:
      - lambda: |-
          uint16_t register_value = x * 10000;
          id(${output_ids[0]}_register).set_value(register_value);

sensor:
  - platform: template
    name: "${friendly_name} ${channel_names[0]} Capacity"
    id: ${output_ids[0]}_capacity
    unit_of_measurement: "%"
    accuracy_decimals: 0
    lambda: |-
      return id(${output_ids[0]}_register).state / 100.0;
    update_interval: 10s

# Repeat for channels 2-4 with registers 0x0002, 0x0003, 0x0004
```

**Alternative if ESPHome doesn't support dynamic loops:**
Create 4 separate channel definitions in the component, parameterized via vars for names/IDs.

### Soggiorno Dual-Fancoil Staging Strategies

**Strategy A - Split (Equal Distribution):**
```yaml
# Both fancoils run at 50% of PID output
# Advantage: Even wear, quieter operation
# Disadvantage: Less granular control at low demand

output:
  - platform: template
    id: soggiorno_cooling_split
    type: float
    write_action:
      - lambda: |-
          // Split PID output 50/50 to both fancoils
          float half_output = x * 0.5;
          id(soggiorno_fc1).set_level(half_output);
          id(soggiorno_fc2).set_level(half_output);
```

**Strategy B - Sequential (Cascading):**
```yaml
# First fancoil 0-100%, second activates when first maxed
# Advantage: More granular control, energy efficient
# Disadvantage: Uneven wear on first unit

output:
  - platform: template
    id: soggiorno_cooling_sequential
    type: float
    write_action:
      - lambda: |-
          if (x <= 0.5) {
            // First 50% of PID: only FC1 active at 0-100%
            id(soggiorno_fc1).set_level(x * 2.0);
            id(soggiorno_fc2).set_level(0.0);
          } else {
            // Second 50% of PID: FC1 at 100%, FC2 at 0-100%
            id(soggiorno_fc1).set_level(1.0);
            id(soggiorno_fc2).set_level((x - 0.5) * 2.0);
          }
```

**Recommendation**: Start with **Sequential** for better low-demand efficiency, can switch to Split via substitution if needed.

### Climate Mode Integration

Fancoils should **only** operate in cooling mode:

```yaml
# In PID climate controller configuration
climate:
  - platform: pid
    id: soggiorno_cooling_pid
    name: "Soggiorno Cooling"
    sensor: soggiorno_temperature_abstracted
    default_target_temperature: 24°C
    cool_output: soggiorno_cooling_output  # Links to staging logic
    
    # Only active in cooling mode
    on_state:
      - if:
          condition:
            lambda: 'return id(modbus_climate_mode).state != 2;'  # Not cooling
          then:
            - climate.control:
                id: soggiorno_cooling_pid
                mode: "OFF"

# Or use automation trigger on climate mode change
on_value:
  - platform: sensor
    sensor_id: modbus_climate_mode
    then:
      - if:
          condition:
            lambda: 'return x != 2;'  # Not cooling mode
          then:
            # Ramp down all fancoils over 5 seconds
            - output.set_level:
                id: soggiorno_fc1
                level: 0
            - output.set_level:
                id: soggiorno_fc2
                level: 0
            # (repeat for cucina and tech_closet)
```

### PID Tuning Guidance

Fancoil control requires **faster response** than floor heating:

```yaml
# Fancoil PID tuning (example for Soggiorno)
climate:
  - platform: pid
    control_parameters:
      kp: 0.5        # Higher than floor heating (more aggressive)
      ki: 0.01       # Moderate integral
      kd: 0.05       # Some derivative for overshoot prevention
      
    deadband_parameters:
      threshold_high: 0.2°C   # Tighter deadband than floor
      threshold_low: -0.2°C
      
    update_interval: 30s  # Faster than floor heating (60s)
```

**Rationale**: Fancoils have much faster thermal response than floor heating, so PID can be more aggressive.
