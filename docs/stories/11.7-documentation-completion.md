## Technical Achievements

### 3-Tier Failover Architecture

**Design Pattern:**
```
Tier 1 (Preferred): UDP from ESP32 room sensor
  ↓ (10s timeout if unavailable)
Tier 2 (Fallback): Home Assistant API
  ↓ (180s timeout if unavailable)
Tier 3 (Emergency): NaN → Emergency shutdown
```

**Benefits:**
- Autonomous operation during HA outages
- Graceful degradation (UDP → HA → Emergency)
- Backward compatible with Epic 5 HA-only (use_udp flag)

### UDP Room Sensor Performance

**Latency Measurements (Pilot Room):**
- UDP broadcast to PID input: 0.4-0.8 seconds (avg 0.6s)

## Lessons Learned

### What Went Well

1. **Pilot Validation (Story 11.5):**
   - Testing Soggiorno first caught WiFi coverage issues early
   - Validated 3-tier failover logic before batch deployment
   - Provided confidence for system-wide rollout

2. **WiFi Survey (Story 11.4):**
   - Pre-deployment survey identified weak signal areas
   - Prevented failed deployments in Camera 6, Locale Tecnico
   - Recommended WiFi extender proactively

3. **Component Reusability:**
   - sensor_failover_3tier.yaml worked for both Epic 9 (supply temps) and Epic 11 (room temps)
   - room_sensors.yaml v6 maintained backward compatibility (use_udp flag)
   - ESP32 template enabled quick configuration of 14 sensors

4. **Incremental Rollout:**
   - Ground floor first (6 rooms) validated before first floor (8 rooms)
   - Staggered enable: pilot → ground floor → first floor minimized risk
   - Easy rollback plan (use_udp flag) provided safety net

### What Could Be Improved

1. **WiFi Coverage:**
   - 2 rooms (Locale Tecnico, Camera 6) have marginal signal (<-70 dBm)
   - **Action:** Add WiFi extender to improve coverage for weak areas
   - **Future:** WiFi survey should be mandatory pre-deployment step

2. **Sensor Placement:**
   - Cucina sensor initially near stove (heat source skewed readings)
   - **Action:** Relocated sensor 2m away, readings normalized
   - **Future:** Deployment guide should emphasize avoiding heat sources

3. **Calibration Drift:**
   - 3 sensors required calibration offset (>±0.5°C error vs reference thermometer)
   - **Action:** Applied temp_offset in YAML, reflashed sensors
   - **Future:** Include calibration step in bench testing before deployment

4. **Documentation Timing:**
   - Hardware BOM created after procurement (should be before ordering)
   - **Action:** Story 11.7 documentation consolidates all knowledge
   - **Future:** Create BOM during planning phase (Epic brief), not post-deployment

## Known Issues and Limitations

### WiFi Coverage Gaps

**Issue:** Locale Tecnico (-72 dBm) and Camera 6 (-73 dBm) have weak WiFi signal  
**Impact:** Packet loss 2.1% and 2.5% respectively (above 1% target)  
**Mitigation:** Both sensors remain operational on Tier 1 (UDP), HA fallback available if needed  
**Resolution:** Install WiFi extender near first floor Camera 6 area (will improve both rooms)

### Humidity UDP Broadcasts (MVP Deferred)

**Issue:** Epic 11 MVP broadcasts temperature only via UDP, humidity still via HA  
**Impact:** Humidity sensing still HA-dependent, not fully autonomous  
**Mitigation:** room_sensors.yaml v6 supports both UDP temp + HA humidity  
**Resolution:** Epic 12 will add humidity UDP broadcasts for complete autonomy

### Same-Subnet Requirement

**Issue:** ESPHome packet_transport requires all devices on same subnet  
**Impact:** Cannot place ESP32 sensors on guest WiFi or separate VLAN  
**Mitigation:** All sensors on main WiFi network (2.4GHz)  
**Resolution:** No workaround available, ESPHome platform limitation

### ESP32 Hardware Limitations

**Issue:** ESP32 supports 2.4GHz WiFi only, no 5GHz support  
**Impact:** Cannot use 5GHz network (less congested, faster)  
**Mitigation:** 2.4GHz adequate for low-bandwidth UDP broadcasts (<1 KB/s)  
**Resolution:** Future hardware upgrade to ESP32-S3 (has 5GHz WiFi)

### Architecture Diagram Update

## Epic 11: UDP Room Sensors (Peer-to-Peer Architecture)

```
┌────────────────────────────────────────────────┐
│ ESP32 Room Sensor (External Device, Per Room) │
│  - Temperature: SHT31 (±0.2°C)                 │
│  - Humidity: SHT31 (±2% RH)                    │
│  - UDP Broadcaster: Port 6053, 10s interval    │
│  - Broadcast IDs: {zone}_room_temp/_humidity   │
└─────────────────┬──────────────────────────────┘
                  │ UDP Broadcast (WiFi 2.4GHz)
                  ▼
┌────────────────────────────────────────────────┐
│ Distribution Board (KC868-A16)                 │
│  - UDP Receiver: packet_transport platform     │
│  - 3-Tier Failover: sensor_failover_3tier.yaml│
│    • Tier 1: UDP sensor (10s timeout)          │
│    • Tier 2: HA sensor (180s timeout)          │
│    • Tier 3: Emergency (NaN → shutdown)        │
│  - Abstracted Sensor: {zone}_room_temp_abstracted│
│  - PID Controller: Uses abstracted sensor      │
└────────────────────────────────────────────────┘
```

**Data Flow:**
1. ESP32 reads SHT31 sensor every 10 seconds
2. ESP32 broadcasts UDP packet (temperature + humidity)
3. Distribution board receives packet, updates UDP sensor
4. sensor_failover_3tier selects active tier (UDP preferred)
5. Abstracted sensor provides value to PID controller
6. PID operates autonomously (no HA dependency)

**Failover Behavior:**
- **Normal:** Tier 1 (UDP) active, <1s latency
- **ESP32 offline:** Tier 2 (HA) active after 10s, still operational
- **Both unavailable:** Tier 3 (Emergency) after 180s, PID forced OFF

**Backward Compatibility:**
- Epic 5 HA-only: `use_udp: false` (default, preserves old behavior)
- Epic 11 UDP mode: `use_udp: true` (enables 3-tier failover)
