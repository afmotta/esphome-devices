## Testing Overview

This guide provides manual test scenarios to validate the coordinator component against the Epic 8 coordinator design specification. All tests verify priority resolution, PID control decisions, and diagnostic observability.

**Prerequisites:**
- ESPHome device with coordinator component included
- Emergency condition component configured
- Window condition component configured
- PID climate component configured
- Access to ESPHome logs (via serial or network)
- Ability to manually set condition global values (via REPL or test automation)

---

## Test Scenario 1: Single Condition Active (Emergency)

**Objective:** Verify coordinator forces PID OFF when only emergency condition is active

**Setup:**
- Initial state: All conditions Normal (state=0)
- PID mode: `HEAT` (running)
- Manually set: `${zone_slug}_emergency_state = 1`

**Steps:**
1. Set emergency condition to Active (state=1)
2. Wait 5 seconds (next coordinator interval)
3. Check ESPHome logs for shutdown message
4. Verify PID mode is OFF
5. Check HA coordinator status sensor

**Expected Results:**
- Log message: `"${zone_name}: SHUTDOWN TRIGGERED - Condition: Emergency (Active), Priority: 1"`
- PID mode: `OFF`
- Coordinator status sensor: `"Shutdown: Emergency (Active)"`

**Pass Criteria:** ✅ PID forced OFF, diagnostic sensor shows Emergency

## Test Scenario 3: Multiple Conditions Active (Priority Resolution)

**Objective:** Verify priority hierarchy when both emergency and window are active

**Setup:**
- Initial state: All conditions Normal
- PID mode: `HEAT` (running)
- Manually set: `${zone_slug}_emergency_state = 1`, `${zone_slug}_window_state = 1`

**Steps:**
1. Set both conditions to Active (state=1) simultaneously
2. Wait 5 seconds (next coordinator interval)
3. Check ESPHome logs for shutdown message
4. Verify log shows "Emergency" (not "Window")
5. Check HA coordinator status sensor

**Expected Results:**
- Log message: `"${zone_name}: SHUTDOWN TRIGGERED - Condition: Emergency (Active), Priority: 1"`
- PID mode: `OFF`
- Coordinator status sensor: `"Shutdown: Emergency (Active)"` (priority 1 < 2)
- Window condition ignored due to lower priority

**Pass Criteria:** ✅ Emergency takes precedence over Window

## Test Scenario 4: Priority Switch During Operation

**Objective:** Verify coordinator switches displayed condition when higher-priority condition clears

**Setup:**
- Initial state: Both conditions Active
- PID mode: OFF (already shut down)
- Coordinator status: `"Shutdown: Emergency (Active)"`

**Steps:**
1. Clear emergency condition (set state=0)
2. Keep window condition Active (state=1)
3. Wait 5 seconds (next coordinator interval)
4. Check ESPHome logs for status change
5. Verify coordinator status sensor updates

**Expected Results:**
- Log message: `"${zone_name}: SHUTDOWN MAINTAINED - Condition: Window (Active), Priority: 2"`
- PID mode: `OFF` (still shut down)
- Coordinator status sensor: `"Shutdown: Window (Active)"` (priority switch)
- Log shows transition from Emergency to Window control

**Pass Criteria:** ✅ Status updates to show Window when Emergency clears

- No PID mode change
- No climate.control API call
- Coordinator recognizes PID already OFF

## Test Scenario 10: Cascading Condition Clearance

**Objective:** Verify coordinator maintains shutdown until ALL conditions clear

**Setup:**
- Initial state: Both conditions Active
- PID mode: OFF

**Steps:**
1. Clear emergency condition (state=0)
2. Wait 5s → Verify status shows "Shutdown: Window (Active)"
3. Clear window condition (state=0)
4. Wait 5s → Verify status shows "Normal (All Clear)"

**Expected Results:**
- T=0s: Status="Shutdown: Emergency (Active)"
- T=5s (emergency clears): Status="Shutdown: Window (Active)"
- T=10s (window clears): Status="Normal (All Clear)"
- PID stays OFF until step 4

**Pass Criteria:** ✅ Coordinator maintains shutdown through condition transitions

## Test Scenario 11: HA Automation Resume (Integration Test)

**Objective:** Verify HA automation can resume PID when coordinator reports clear

**Setup:**
- HA automation configured to monitor coordinator_status sensor
- All conditions Active → coordinator status = "Shutdown: Emergency (Active)"
- PID mode: OFF

**Steps:**
1. Clear all conditions (state=0)
2. Wait 5s → Verify coordinator status = "Normal (All Clear)"
3. Wait 30s (HA automation delay)
4. Verify HA automation triggers
5. Verify PID mode set to desired state (e.g., HEAT)

**Expected Results:**
- Coordinator status: `"Normal (All Clear)"`
- HA automation triggers after 30s delay
- PID mode set by HA (e.g., to `HEAT`)
- Coordinator does not interfere with HA-driven resume

**Pass Criteria:** ✅ HA automation successfully resumes climate control

## Coordinator Algorithm Compliance Checklist

After completing manual tests, verify coordinator meets design specification:

### Priority Resolution
- [ ] Reads all condition states and priorities (Test 1-3)
- [ ] Filters conditions where state > 0 (Test 6)
- [ ] Selects lowest priority number among active conditions (Test 3)
- [ ] Handles simultaneous triggers deterministically (Test 3)
- [ ] Switches active condition when higher priority clears (Test 4)

### PID Control Logic
- [ ] Forces PID OFF when any condition Active (Test 1-2)
- [ ] Forces PID OFF when any condition Recovering (Test 5)
- [ ] Maintains PID OFF when already shut down (Test 7)
- [ ] Does NOT auto-resume PID when all conditions clear (Test 6)

### Edge Cases
- [ ] Validates priority range (1-99), ignores invalid (Test 8)
- [ ] Handles missing globals gracefully (Test 9)
- [ ] Maintains shutdown through condition transitions (Test 10)

### Observability
- [ ] Diagnostic sensor shows correct status for all states (All tests)
- [ ] Logs shutdown triggers at WARN level (Test 1-2)
- [ ] Logs shutdown maintenance at DEBUG level (Test 7)
- [ ] Logs normal state at VERBOSE level (Test 6)
- [ ] Includes condition name, state, and priority in logs (All tests)

### Integration
- [ ] Works with HA automation for resume (Test 11)
- [ ] Coexists with condition components (All tests)
- [ ] 5-second polling interval acceptable (All tests)

## Known Limitations

1. **5-second latency:** Coordinator polls every 5s, max detection delay is 10s (condition interval + coordinator interval)
2. **MVP: No auto-resume:** Coordinator does not automatically resume PID, relies on HA automation
3. **Static condition list:** MVP hardcodes emergency + window, future versions will support dynamic registration
4. **No mode-awareness:** Coordinator doesn't understand PID modes, just forces OFF

## Troubleshooting

**Issue:** Coordinator status stuck on old condition after it clears
- Check: Condition global state is actually 0
- Check: Coordinator interval is running (check logs)
- Restart: ESPHome device to reset state

**Issue:** PID not forced OFF when condition Active
- Check: PID entity ID matches `pid_id` var
- Check: ESPHome logs show "SHUTDOWN TRIGGERED" message
- Check: Climate API call succeeds (no errors)

**Issue:** Coordinator status shows "Unknown" or blank
- Check: All condition components included in device YAML
- Check: Condition globals exposed with correct names
- Restart: ESPHome device

**Issue:** HA automation doesn't resume PID
- Check: Automation entity_id matches coordinator status sensor
- Check: Automation `for: "00:00:30"` delay isn't too long
- Check: Automation action has correct climate entity_id
