### Pilot Room Selection: Why Soggiorno?

**Reasons for choosing Soggiorno as pilot:**
1. **High visibility:** Living room = high-traffic area, easy to monitor sensor
2. **Good WiFi signal:** -55 dBm (Story 11.4 survey), reliable UDP broadcasts
3. **Existing Epic 5 deployment:** Can compare UDP vs HA-only performance
4. **Representative room:** Radiant + fancoil equipped (typical ground floor zone)
5. **Easy access:** Ground floor, convenient for troubleshooting/re-flashing

### Testing Procedure Details

**UDP Latency Test (Expected <1 second):**

ESP32 sensor logs:
```
[23:45:00.123] [sensor:456] 'soggiorno_temperature': Sending state 22.3°C
[23:45:00.234] [packet_transport:789] Broadcasting soggiorno_room_temp = 22.3
```

Distribution board logs (same machine time):
```
[23:45:00.345] [packet_transport:123] Received packet from soggiorno_room_sensor
[23:45:00.456] [sensor:789] 'soggiorno_udp_temp': Got value 22.3°C
[23:45:00.567] [template:234] 'soggiorno_room_temp_abstracted': 22.3°C (Tier: UDP)
```

**Latency calculation:** 0.567 - 0.123 = 0.444 seconds (excellent, well below 1s target)

**HA Restart Test (Zero Blackout Expected):**

Timeline:
1. **T=0s:** HA restart initiated, sensor tier = "UDP"
2. **T=10s:** HA services stopping, distribution board logs show HA API connection lost
3. **T=20s:** HA rebooting, sensor tier still "UDP" (no failover needed)
4. **T=30s:** HA services starting, ESPHome integration reconnecting
5. **T=60s:** HA fully operational, sensor tier still "UDP" (never failed over)

**Success criteria:** PID continues operating throughout HA restart, no emergency shutdown, abstracted sensor remains stable.

**UDP Failure Test (30-Second Failover Expected):**

Timeline:
1. **T=0s:** ESP32 powered off, last UDP broadcast received
2. **T=10s:** UDP timeout triggered (3 missed broadcasts)
3. **T=11s:** Failover to Tier 2, logs: "UDP unavailable, switching to HA tier"
4. **T=12s:** Abstracted sensor now reads from HA sensor
5. **T=13s:** Diagnostic sensor updates: "UDP" → "HA"
6. **T=30s:** PID continues operating on HA sensor (no interruption)

**Success criteria:** Failover completes within 30 seconds, PID continues operating, no emergency shutdown.

**Emergency Shutdown Test (180-Second Timeout Expected):**

Timeline:
1. **T=0s:** HA stopped (UDP already down from previous test), sensor tier = "HA"
2. **T=30s:** HA timeout triggered, logs: "HA sensor unavailable"
3. **T=180s:** Emergency timeout triggered (Story 11.1 sensor_failover_3tier)
4. **T=181s:** Failover to Tier 3, logs: "Emergency condition: both UDP + HA unavailable"
5. **T=182s:** Diagnostic sensor updates: "HA" → "Emergency"
6. **T=183s:** Coordinator forces PID OFF (Epic 8 emergency condition integration)

**Success criteria:** Emergency shutdown triggers after 180s, PID forced OFF, system safe.

**UDP Recovery Test (10-Second Recovery Expected):**

Timeline:
1. **T=0s:** ESP32 powered on, WiFi connecting
2. **T=5s:** WiFi connected, first UDP broadcast sent
3. **T=6s:** Distribution board receives broadcast, UDP sensor updates
4. **T=7s:** Failover detects UDP available, logs: "UDP recovered, switching to UDP tier"
5. **T=8s:** Abstracted sensor now reads from UDP sensor
6. **T=9s:** Diagnostic sensor updates: "Emergency" → "UDP"
7. **T=10s:** Coordinator detects emergency cleared, allows PID control

**Success criteria:** Recovery completes within 10 seconds, PID resumes operation, system autonomous again.

### UDP Packet Loss Calculation

**Formula:**
```
Expected broadcasts = (48 hours × 3600 seconds/hour) ÷ 10 seconds/broadcast
                    = 17,280 broadcasts

Actual broadcasts = Count from distribution board logs or diagnostic sensor

Packet loss % = (Expected - Actual) ÷ Expected × 100%
```

**Acceptable packet loss:** <1% (i.e., >17,100 broadcasts received)

**If packet loss >1%:**
- Check WiFi signal strength (may have degraded)
- Check router load (other devices causing interference)
- Relocate sensor or add WiFi extender

### Troubleshooting Common Issues

**Issue: Sensor tier shows "HA" instead of "UDP" after enable**
- **Cause:** ESP32 not broadcasting or distribution board not receiving
- **Debug:** Check ESP32 logs for broadcast messages
- **Debug:** Run tcpdump on distribution board WiFi
- **Fix:** Verify ESP32 WiFi connected, same subnet as distribution board

**Issue: Sensor tier flaps between "UDP" and "HA"**
- **Cause:** Intermittent WiFi connectivity or high packet loss
- **Debug:** Check WiFi signal strength (<-70 dBm = weak)
- **Fix:** Relocate sensor closer to router or add WiFi extender

**Issue: PID emergency shutdown during HA restart**
- **Cause:** UDP mode not actually enabled (still using Epic 5 HA-only)
- **Debug:** Check distribution board logs for "UDP sensor unavailable" (should NOT appear)
- **Fix:** Verify `use_udp: true` in room_control include, recompile/flash

**Issue: UDP recovery takes >10 seconds**
- **Cause:** Broadcast interval too long or failover timeout too conservative
- **Debug:** Check `update_interval` in ESP32 sensor config (should be 10s)
- **Fix:** Reduce `update_interval` to 5s for faster recovery (increases bandwidth slightly)

**Issue: Flash usage >80% after UDP receiver added**
- **Cause:** Too many UDP providers or diagnostic sensors
- **Fix:** Remove unused components (e.g., deprecated room_sensors v4)
- **Fix:** Disable unused binary sensors or text sensors
