### PID Controller Integration

1. Update PID controllers to use room temperature instead of supply temperature:
   - PID setpoint = desired room temperature (user-configurable)
   - PID sensor input = room temperature (from room_sensors component)
   - Supply temperature becomes monitored value (not control input)
   - Mixing valve adjusts supply temp to achieve room temp setpoint

2. Implement sensor failover for room temperature:
   - Primary: Local room sensor (Modbus or 1-Wire/I2C)
   - Fallback: Home Assistant room sensor (existing `sensor.termometro_*_temperature`)
   - Emergency: Supply temperature as proxy (degraded control mode)

3. Supply temperature monitoring and limits:
   - Track supply temperature for safety (prevent overheating)
   - Implement maximum supply temperature limit (configurable, default: 45°C heat, 15°C cool)
   - Log warnings if supply temp exceeds safe range

### Home Assistant Integration

1. Room sensor data exposed to Home Assistant:
   - Temperature sensors: `sensor.{zone}_room_temperature`
   - Humidity sensors: `sensor.{zone}_room_humidity`
   - Sensor status: `binary_sensor.{zone}_room_sensor_online`
   - Last update timestamp: `sensor.{zone}_room_sensor_last_update`

2. Diagnostic entities:
   - If Modbus: Communication error count per sensor
   - If 1-Wire: Bus scan errors and CRC failures
   - Sensor battery level (if applicable to sensor model)

3. Climate entity updates:
   - Current temperature reflects room temperature
   - Supply temperature as separate attribute
   - Humidity displayed as climate entity attribute

## Dev Notes

### Architecture Context

This story implements **room-level temperature and humidity sensing** for precise zone control. Previously, PID controllers used supply temperature (from mixing valve Dallas sensors) as their control input. With room sensors, PID controllers can now target actual room temperature, with supply temperature automatically adjusted to meet room demand.

**System Context:**
- **Current Control**: PID setpoint = supply temperature → limited accuracy, slow response
- **Enhanced Control**: PID setpoint = room temperature → direct comfort control, faster response
- Supply temperature becomes monitored value for safety limits, not control input

**Benefits:**
1. **Improved Accuracy**: Control room temperature directly, not supply temperature proxy
2. **Better Comfort**: Account for room-specific factors (sun exposure, occupancy, air circulation)
3. **Humidity Monitoring**: Foundation for future humidity-aware automation
4. **Per-Zone Tuning**: Different rooms can have different control characteristics

### Technology Selection Decision

**Critical Decision Point**: This story requires selecting between two sensor technology approaches:

#### Option A: Modbus Temperature/Humidity Sensors

**Advantages:**
- Standardized protocol (same as master/slave communication)
- Easy to add/remove sensors (just assign Modbus address)
- Professional-grade sensors available (industrial accuracy)
- Long cable runs supported (RS485 allows up to 1200m)
- Digital communication (no analog noise issues)

**Disadvantages:**
- Higher cost per sensor (~$30-50 per sensor vs ~$5-10 for 1-Wire)
- Additional Modbus devices on bus (4 sensors = addresses 10-13)
- Increases master polling cycle time
- More complex wiring if sensors distributed across house

**Implementation:**
```yaml
# Modbus room sensor (master polls, slave reads from master)
modbus_controller:
  - id: room_sensor_soggiorno
    address: 10  # Modbus slave address
    modbus_id: rs485_bus

sensor:
  - platform: modbus_controller
    modbus_controller_id: room_sensor_soggiorno
    id: soggiorno_room_temp_modbus
    name: "Soggiorno Room Temperature"
    address: 0x0001
    register_type: holding
    value_type: S_WORD
    accuracy_decimals: 2
    filters:
      - multiply: 0.01  # Scale from ×100
```

#### Option B: 1-Wire Temperature + I2C Humidity Sensors

**Advantages:**
- Lower cost (~$3 for DS18B20, ~$5-10 for DHT22/SHT3x)
- No additional Modbus bus loading
- Local sensors on slave board (no master dependency)
- Simple wiring (1-Wire bus or I2C bus)
- Proven ESPHome integration

**Disadvantages:**
- Mixed technologies (1-Wire + I2C)
- Limited humidity sensor options for 1-Wire
- Shorter cable runs (1-Wire: ~100m, I2C: ~5m)
- Bus scanning overhead for multiple sensors
- Less professional/industrial appearance

**Implementation:**
```yaml
# 1-Wire temperature + I2C humidity
dallas:
  - pin: GPIO32
    update_interval: 30s

sensor:
  - platform: dallas
    address: 0x1234567890ABCDEF
    id: soggiorno_room_temp_dallas
    name: "Soggiorno Room Temperature"

  - platform: sht3xd  # or dht22
    address: 0x44
    temperature:
      id: soggiorno_room_temp_i2c
      internal: true
    humidity:
      id: soggiorno_room_humidity
      name: "Soggiorno Room Humidity"
    update_interval: 30s
```

**Recommendation Process:**
1. Document both approaches in `docs/sensor-technology-selection.md`
2. Consider budget constraints (4 zones × sensor cost)
3. Consider installation complexity (wiring access)
4. Consider future expansion plans (more zones = more sensors)
5. Test one zone with each approach during development
6. Make final selection based on testing results

**Default Selection (if no preference)**: Option B (1-Wire + I2C) due to lower cost and simpler integration. Modbus Option A is better for larger installations or professional/commercial deployments.

### PID Control Architecture Change

**Current Architecture:**
```
Dallas Supply Temp → PID Controller → Mixing Valve → Supply Water → Room
                     ↑ Setpoint
```

**Enhanced Architecture:**
```
Room Sensor → PID Controller → Mixing Valve → Supply Water → Room
                               ↑ Feedback
Dallas Supply Temp → Safety Monitor (limits, warnings)
```

**Key Changes:**
1. **PID Sensor Input**: Changes from supply temp to room temp
2. **PID Setpoint**: Changes from supply temp target to room temp target
3. **Supply Temp Role**: Monitoring only, not control input
4. **Mixing Valve Control**: Adjusts supply temp to achieve room setpoint

**Example Configuration Change:**

**Before (Story 1.5 and earlier):**
```yaml
climate:
  - platform: pid
    id: soggiorno_floor_heat_pid
    name: "Soggiorno Floor Heat"
    sensor: dallas_supply_temp_soggiorno  # Supply temperature
    heat_output: soggiorno_heat_output
    control_parameters:
      kp: 0.5
      ki: 0.001
      kd: 0.01
    default_target_temperature: 35°C  # Supply temp target
```

**After (Story 1.6):**
```yaml
climate:
  - platform: pid
    id: soggiorno_floor_heat_pid
    name: "Soggiorno Floor Heat"
    sensor: soggiorno_room_temp_abstracted  # Room temperature
    heat_output: soggiorno_heat_output
    control_parameters:
      kp: 0.8  # May need retuning
      ki: 0.005
      kd: 0.05
    default_target_temperature: 22°C  # Room temp target

sensor:
  - platform: template
    id: soggiorno_supply_temp_monitor
    name: "Soggiorno Supply Temp Monitor"
    lambda: 'return id(dallas_supply_temp_soggiorno).state;'
    
automation:
  - id: supply_temp_safety_check
    trigger:
      - platform: template
        value_template: >
          {{ id(soggiorno_supply_temp_monitor).state > 45.0 }}
    action:
      - logger.log: "WARNING: Supply temp exceeds safe limit"
      - climate.control:
          id: soggiorno_floor_heat_pid
          mode: OFF
```

**PID Retuning Considerations:**
- Room temperature responds **slower** than supply temperature (thermal mass)
- May need to increase `ki` (integral) to handle steady-state error
- May need to increase `kd` (derivative) to prevent overshoot
- Start with conservative values and tune during testing

### Sensor Failover Architecture

**Three-Tier Failover for Room Temperature:**

```yaml
sensor:
  - platform: template
    id: soggiorno_room_temp_abstracted
    name: "Soggiorno Room Temperature"
    device_class: temperature
    unit_of_measurement: "°C"
    accuracy_decimals: 1
    update_interval: 10s
    lambda: |-
      // TIER 1: Local room sensor (Modbus or 1-Wire/I2C)
      if (id(soggiorno_room_sensor_local).has_state()) {
        auto age = id(soggiorno_room_sensor_local).get_last_update_timestamp();
        if (millis() - age < 30000) {  // Less than 30s old
          return id(soggiorno_room_sensor_local).state;
        }
      }
      
      // TIER 2: Home Assistant room sensor (fallback)
      if (id(ha_termometro_soggiorno).has_state()) {
        return id(ha_termometro_soggiorno).state;
      }
      
      // TIER 3: Supply temperature proxy (degraded mode)
      // Use supply temp with offset as estimate
      if (id(dallas_supply_temp_soggiorno).has_state()) {
        // Estimate room temp = supply temp - 10°C (rough approximation)
        return id(dallas_supply_temp_soggiorno).state - 10.0;
      }
      
      // TIER 4: Emergency (no data available)
      return NAN;

text_sensor:
  - platform: template
    id: soggiorno_temp_source
    name: "Soggiorno Temperature Source"
    lambda: |-
      if (id(soggiorno_room_sensor_local).has_state()) {
        return {"Local Room Sensor"};
      } else if (id(ha_termometro_soggiorno).has_state()) {
        return {"Home Assistant Fallback"};
      } else if (id(dallas_supply_temp_soggiorno).has_state()) {
        return {"Supply Temp Proxy (Degraded)"};
      } else {
        return {"No Sensor Available (Emergency)"};
      }
```

**Failover Behavior:**
- **Tier 1**: Local room sensor (best accuracy, fastest response)
- **Tier 2**: Home Assistant room sensor (proven fallback)
- **Tier 3**: Supply temperature proxy (degraded but functional control)
- **Tier 4**: NAN → Triggers emergency shutdown (from Story 1.4)

**Master Polling Sequence (If Modbus Sensors):**
1. Poll slave address 2 (ground floor A16) - status registers
2. Poll slave address 3 (first floor A16) - status registers
3. Poll room sensor address 10 (Soggiorno) - temp/humidity
4. Poll room sensor address 11 (Cucina) - temp/humidity
5. Poll room sensor address 12 (Bagno) - temp/humidity
6. Poll room sensor address 13 (Anticamera) - temp/humidity
7. Write room data to master registers 400-407
8. Slaves read registers 400-407 for local room sensor data
9. Repeat cycle every 30 seconds

### Critical Success Factors

1. **Technology Selection**: Wrong choice impacts cost, reliability, maintainability
2. **Sensor Placement**: Poor placement (near heat sources, poor air circulation) = bad data
3. **PID Retuning**: Room temperature response is different from supply temp
4. **Failover Logic**: Must handle sensor failures gracefully without comfort disruption
5. **Supply Temp Limits**: Safety monitoring critical to prevent overheating/undercooling
6. **Accuracy Verification**: Room sensors must be calibrated (±0.3°C) for good control
7. **Bus Loading**: If Modbus sensors, must verify polling cycle stays within 500ms

### Known Limitations and Future Work

**Current Limitations:**
- Room sensors on ground floor only (first floor uses supply temp control)
- No automatic humidity-based control (monitoring only)
- PID auto-tuning not implemented (manual tuning required)
- Supply temperature proxy mode is rough estimate (not precise control)

**Future Enhancements (Beyond Epic 1):**
- Extend room sensors to first floor and second floor
- Implement humidity-aware control (adjust supply temp based on humidity)
- Dew point calculation for condensation prevention
- Automatic PID parameter adjustment based on room characteristics
- Machine learning for predictive control (anticipate temperature changes)
- Occupancy detection integration (adjust setpoints based on occupancy)
