**Key Design Decisions:**
- **Mode-awareness retained:** Window condition keeps mode checking logic (not "dumb sensor") because trigger semantics are mode-dependent (heating vs cooling vs OFF)
- **Two paths to Recovering:** Active→Recovering can trigger via (1) window close OR (2) mode change to non-shutdown mode
- **Grace period visibility:** Text sensor shows "Normal (window open Xs)" during timeout grace period for user awareness
- **Graceful degradation:** Unavailable window sensor treated as "closed" (fail-safe for energy efficiency)

## Acceptance Criteria

### Functional Requirements

1. **State Global Exposed:**
   - Global `${zone_slug}_window_state` defined as `int` type
   - Initial value is `0` (Normal)
   - Transitions: Normal(0) → Active(1) when window open 180s AND mode matches `window_shutdown_modes`
   - Transitions: Active(1) → Recovering(2) when window closes OR mode changes to non-shutdown mode
   - Transitions: Recovering(2) → Normal(0) after 60s stability
   - Transitions: Recovering(2) → Active(1) if window re-opens during recovery (and mode still matches)

2. **Priority Global Exposed:**
   - Global `${zone_slug}_window_priority` defined as `int` type
   - Fixed value: `2` (second highest priority, below emergency)
   - Value set at initialization, never changes at runtime

3. **Window Detection Logic:**
   - Monitors HA window sensor via `entity_id: ${ha_window_sensor_id}`
   - Tracks last window-open timestamp
   - Validates sensor state is "on" (open) or "off" (closed)
   - Triggers after 180s (configurable via `window_timeout` var)

4. **Mode-Awareness Logic:**
   - Reads PID current mode from `id(${pid_id})->mode`
   - Compares against CSV list `window_shutdown_modes` (e.g., "cooling, heating")
   - Only triggers if: window open AND current mode in shutdown modes list
   - Clears Active state if mode changes out of shutdown modes (even if window still open)

5. **State Machine Behavior:**
   - Interval polling every 10s to check window + mode status
   - Implements all state transitions per interface spec
   - Logs state transitions at INFO/WARN levels
   - Includes zone name and reason in log messages (e.g., "window open in cooling mode")

6. **Diagnostic Text Sensor:**
   - Entity ID: `${zone_slug}_window_state_sensor`
   - Name: `"${zone_name} Window State"`
   - Updates every 5s
   - Displays: "Normal" / "Active" / "Recovering" / "Unknown"

### Integration Requirements

7. **Interface Contract Compliance:**
   - Passes all checklist items in `epic-8-condition-interface-spec.md`
   - State global naming matches spec exactly
   - Priority global in valid range (1-99)
   - No direct PID control (coordinator responsibility)

8. **Backward Compatibility:**
   - Component can coexist with Epic 7 `room_window_detection.yaml` during migration
   - Uses unique global names (no conflicts with Epic 7 globals)

9. **Room Integration:**
   - Component accepts same vars as Epic 7:
     - `zone_slug`, `zone_name`, `ha_window_sensor_id`, `pid_id`, `window_shutdown_modes`
   - Adds optional vars: `window_timeout` (default: 180), `recovery_timeout` (default: 60)

### Quality Requirements

10. **Code Quality:**
    - Header comment documents purpose, interface version, and required vars
    - Mode-awareness logic clearly commented
    - CSV parsing for shutdown modes list is robust
    - C++ lambda code follows ESPHome best practices

11. **Testing Validation:**
    - Manual test: Open window in heating mode (if "heating" in shutdown modes) → verify Active after 180s
    - Manual test: Open window in OFF mode → verify stays Normal
    - Manual test: Switch mode from heating to OFF while window open → verify transitions to Recovering
    - Manual test: Close window → verify Recovering immediately → Normal after 60s

## Technical Notes

**Implementation Approach:**

1. Copy `components/room_window_detection.yaml` to `components/room_window_condition.yaml`
2. Add `condition_id` and `priority_value` to defaults section
3. Rename globals from Epic 7 names to Epic 8 contract names
4. Remove PID control logic (all `auto call = id(${pid_id})->make_call()` blocks)
5. Refactor state machine lambda to use tri-state enum (0/1/2)
6. Keep mode-awareness logic (window condition remains "smart")
7. Update text sensor to read from `_window_state` global
8. Add optional debugging globals (`last_trigger_time`)

**Mode-Awareness Design Decision:**

Per brainstorming session, window condition keeps mode-awareness logic for two reasons:
1. **Semantic correctness:** Window doesn't cause emergency in heating mode (waste vs. safety)
2. **Condition autonomy:** Condition understands its own trigger semantics
3. **Coordinator simplicity:** Coordinator doesn't need mode-specific rule engine

This is a deviation from "dumb sensor" principle but justified for domain logic.

**Existing Pattern Reference:**

- Epic 7 mode-awareness in `components/room_window_detection.yaml` (lines 80-120)
- Epic 7 state machine in `components/room_window_detection.yaml` (lines 40-70)
- CSV parsing patterns in ESPHome (split on comma, trim whitespace)

**Key Constraints:**

- Must NOT control PID directly (coordinator responsibility)
- Must NOT read emergency condition state (coordinator responsibility)
- Must use exact global naming convention from interface spec
- Mode-awareness must check ALL modes in CSV list
