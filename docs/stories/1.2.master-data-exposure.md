## Story
**As a** master controller (gruppo-miscelazione),  
**I want** to expose my Dallas temperature sensors and coordination data via Modbus registers,  
**so that** slave boards can read supply temperatures and mode settings without Home Assistant.

   - Register 300: Master heartbeat/timestamp
3. Master responds correctly to slave read requests (function code 0x03)
4. Register values update in real-time as Dallas sensors report new temperatures
5. Documentation created: `docs/modbus-register-map.md` with master register definitions
6. Master status sensor in HA shows active polling state

### Modbus Register Map Design

**Master (A6, Address 1) — Holding Registers (Read by Slaves):**

| Register     | Name                      | Type   | Description                     | Scaling       |
| ------------ | ------------------------- | ------ | ------------------------------- | ------------- |
| 0x0064 (100) | `dallas_piano_terra_temp` | INT16  | Ground floor supply temperature | ×100 (0.01°C) |
| 0x00C8 (200) | `climate_mode`            | ENUM   | Heat=1, Cool=2, Off=0           | Direct value  |
| 0x012C (300) | `master_heartbeat`        | UINT16 | Heartbeat counter/timestamp     | Increments    |

**Why these registers:**
- Register 100: Supply temperature for ground floor (Piano Terra) mixing valve
- Register 200: Climate mode coordination for all zones (heat/cool/off)
- Register 300: Master liveness indicator (slaves can detect master failures)

**Temperature Scaling:** 
- Dallas sensors report float values (e.g., 23.45°C)
- Modbus registers are 16-bit integers
- Multiply by 100: 23.45 → 2345 (preserves 0.01°C precision)
- Slaves divide by 100 to recover float: 2345 → 23.45

### Dallas Sensor Integration

**Current Dallas Sensor Configuration:**
The gruppo-miscelazione device has Dallas DS18B20 temperature sensors on 1-Wire buses (GPIO32, GPIO33). These sensors are addressed by ROM ID:
- Ground floor supply: `dallas_0x81000000b3e6f628`
- First floor supply (future): `dallas_0xe9000000b366ed28`

**Current Sensor Usage:**
- Sensors update every 60 seconds (existing behavior)
- PID controllers use these sensors directly for mixing valve control
- Sensors also report to Home Assistant via ESPHome API

**New Modbus Integration:**
- Add 10-second interval script to sync sensor values to Modbus registers
- This creates a faster update rate for Modbus (10s vs 60s sensor updates)
- Slaves poll registers every 10s, getting most recent sensor value

### Climate Mode Synchronization

**Current Climate Mode Source:**
The master currently reads climate mode from Home Assistant:
```yaml
sensor:
  - platform: homeassistant
    id: master_thermostat_mode
    entity_id: sensor.thermostat_mode
    # Returns: "heat" | "cool" | "off"
```

**Modbus Climate Mode Mapping:**
We need to convert text values to numeric for Modbus:
- "heat" → 1
- "cool" → 2  
- "off" → 0
- Any other value → 0 (safe default)

**Implementation Pattern:**
```yaml
interval:
  - interval: 10s
    then:
      - lambda: |-
          auto mode_str = id(master_thermostat_mode).state;
          if (mode_str == "heat") {
            id(modbus_reg_climate_mode).publish_state(1);
          } else if (mode_str == "cool") {
            id(modbus_reg_climate_mode).publish_state(2);
          } else {
            id(modbus_reg_climate_mode).publish_state(0);
          }
```

This allows slaves to switch modes without Home Assistant connectivity.

### ESPHome Modbus Register Implementation

**Modbus Number Platform for Register Exposure:**
```yaml
number:
  - platform: modbus_controller
    modbus_controller_id: modbus_master
    id: modbus_reg_dallas_piano_terra
    address: 0x0064  # Register 100
    value_type: S_WORD  # Signed 16-bit word
    multiply: 100  # Scaling factor applied by ESPHome
    internal: true  # Not exposed to HA (only via Modbus)
```

**Key Configuration Points:**
- `platform: modbus_controller` — Uses the modbus_controller from Story 1.1
- `modbus_controller_id` — Reference to master controller ID
- `address` — Hexadecimal register address (0x0064 = 100 decimal)
- `value_type: S_WORD` — Signed word for temperatures (can be negative)
- `value_type: U_WORD` — Unsigned word for mode/heartbeat (always positive)
- `internal: true` — Register not exposed to Home Assistant directly
- `multiply` — ESPHome applies scaling automatically

**Syncing Sensor to Register:**
```yaml
interval:
  - interval: 10s
    then:
      - lambda: |-
          id(modbus_reg_dallas_piano_terra).publish_state(
            id(dallas_0x81000000b3e6f628).state * 100
          );
```

### Component Reusability and Parameterization

**vars: Pattern for Flexibility:**
The component should support parameterization to handle multiple sensors/registers:

```yaml
# In components/modbus_master.yaml
defaults:
  vars:
    ground_floor_sensor_id: "dallas_0x81000000b3e6f628"
    ground_floor_register: 0x0064
    climate_mode_sensor_id: "master_thermostat_mode"
    climate_mode_register: 0x00C8
    heartbeat_register: 0x012C
```

**Usage in device YAML:**
```yaml
packages:
  modbus_master: !include
    file: ../components/modbus_master.yaml
    vars:
      ground_floor_sensor_id: "dallas_0x81000000b3e6f628"
      ground_floor_register: 0x0064
```

This allows future expansion (e.g., adding first floor sensor to different register).

[Source: architecture.md#2.3-software-architecture-patterns]

### Feature Flag Integration

**CRITICAL:** Component must respect the `use_modbus` flag:

```yaml
# Only sync to registers if Modbus is enabled
interval:
  - interval: 10s
    then:
      - if:
          condition:
            lambda: 'return id(use_modbus) == "true";'
          then:
            - lambda: |-
                id(modbus_reg_dallas_piano_terra).publish_state(
                  id(dallas_0x81000000b3e6f628).state * 100
                );
```

When `use_modbus: false`, registers are not updated (saves CPU cycles, allows safe testing).
