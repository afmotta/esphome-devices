## User Story

As a **system owner**,  
I want **the emergency shutdown detection logic refactored into a standalone condition component that exposes state+priority via the Epic 8 interface contract**,  
So that **the coordinator can read emergency state and coordinate with other conditions through a standardized interface**.

---

## Story Context

**Existing System Integration:**

- **Refactors from:** `components/room_sensors.yaml` (Epic 5 detection logic)
- **Deprecates:** `components/room_emergency_shutdown.yaml` (control logic moves to coordinator)
- **Technology:** ESPHome YAML, C++ lambdas, Home Assistant sensor integration
- **Follows pattern:** Epic 5 3-state machine (Normal → Active → Recovering) with 180s/60s timeouts
- **Touch points:** 
  - Home Assistant temperature sensor via `sensor.homeassistant` platform
  - Global variables for state communication
  - Text sensor for HA dashboard visibility

**Key Changes from Epic 5:**

- **Remove:** PID control logic (moves to coordinator)
- **Add:** Priority global (value=1, highest priority)
- **Standardize:** Global naming to `${zone_slug}_emergency_state` and `${zone_slug}_emergency_priority`
- **Retain:** All detection logic, timeout management, state machine transitions

## Acceptance Criteria

### Functional Requirements

1. **State Global Exposed:**
   - Global `${zone_slug}_emergency_state` defined as `int` type
   - Initial value is `0` (Normal)
   - Transitions: Normal(0) → Active(1) when sensor unavailable 180s
   - Transitions: Active(1) → Recovering(2) when sensor restored
   - Transitions: Recovering(2) → Normal(0) after 60s stability
   - Transitions: Recovering(2) → Active(1) if sensor fails again during recovery

2. **Priority Global Exposed:**
   - Global `${zone_slug}_emergency_priority` defined as `int` type
   - Fixed value: `1` (highest priority)
   - Value set at initialization, never changes at runtime

3. **Sensor Detection Logic:**
   - Monitors HA temperature sensor via `entity_id: ${ha_temperature_sensor_id}`
   - Tracks last valid sensor timestamp via `on_value` trigger
   - Validates sensor data is not NaN
   - Triggers emergency after 180s (configurable via `emergency_timeout` var)

4. **State Machine Behavior:**
   - Interval polling every 10s to check sensor status
   - Implements all state transitions per interface spec
   - Logs state transitions at INFO/WARN levels
   - Includes zone name in log messages

5. **Diagnostic Text Sensor:**
   - Entity ID: `${zone_slug}_emergency_state_sensor`
   - Name: `"${zone_name} Emergency State"`
   - Updates every 5s
   - Displays: "Normal" / "Active" / "Recovering" / "Unknown"

### Integration Requirements

6. **Interface Contract Compliance:**
   - Passes all checklist items in `epic-8-condition-interface-spec.md`
   - State global naming matches spec exactly
   - Priority global in valid range (1-99)

7. **Backward Compatibility:**
   - Component can coexist with Epic 5 `room_emergency_shutdown.yaml` during migration
   - Uses unique global names (no conflicts with Epic 5 globals)

8. **Room Integration:**
   - Component accepts same vars as Epic 5 `room_sensors.yaml`:
     - `zone_slug`, `zone_name`, `ha_temperature_sensor_id`
   - Adds optional vars: `emergency_timeout` (default: 180), `recovery_timeout` (default: 60)

### Quality Requirements

9. **Code Quality:**
   - Header comment documents purpose, interface version, and required vars
   - C++ lambda code follows ESPHome best practices
   - Defaults section provides sensible fallback values
   - Comments explain state transition logic

10. **Testing Validation:**
    - Manual test: Disconnect HA sensor → verify Active after 180s
    - Manual test: Reconnect sensor → verify Recovering immediately → Normal after 60s
    - Manual test: Disconnect during Recovering → verify re-triggers to Active

11. **Documentation:**
    - Component header references `epic-8-condition-interface-spec.md v1.0`
    - Inline comments explain non-obvious logic
    - Example usage in component or migration docs
